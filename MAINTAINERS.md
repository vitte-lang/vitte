## Decision-Making

### Consensus

Most decisions are made by consensus among active maintainers:
- Open discussion on GitHub issues
- Everyone has voice in design decisions
- Preference for consensus, can vote if needed

### Major Changes

Large changes (breaking changes, major features) require:
- RFC (Request for Comments) discussion
- Approval from at least 2 core maintainers
- Community feedback period (1 week minimum)

### Release Decisions

Releases are decided by the project lead with input from maintainers:
- Version bumping follows [Semantic Versioning](https://semver.org)
- Release notes prepared by core team
- Announcement on social media and community channels

## Responsibilities

### All Maintainers

- Review pull requests
- Triage issues
- Respond to community questions
- Maintain code quality
- Follow Code of Conduct

### Release Manager

- Coordinate releases
- Prepare changelogs
- Build and test artifacts
- Publish releases
- Post announcements

### Steering Committee

- Meet quarterly to discuss strategy
- Set roadmap priorities
- Resolve conflicts
- Approve major changes

## Onboarding New Maintainers

New maintainers are added by:

1. **Nomination** - Existing maintainer nominates contributor
2. **Discussion** - Review with current maintainers
3. **Consensus** - Approval by majority of maintainers
4. **Announcement** - Welcome to the team

## Stepping Down

Maintainers can step down voluntarily:

1. **Notify project lead** - Give notice and reason
2. **Transition** - Help train replacement if possible
3. **Archive contributions** - Ensure knowledge is preserved

## Repository Access

- **Admin**: Project lead (merge authority, settings)
- **Maintain**: Core maintainers (PR merging, issue closing)
- **Triage**: Active contributors (label management, basic moderation)
- **Write**: Regular contributors (branch management)

## Communication

### Channels

- **Issues & PRs**: GitHub (public discussions)
- **Announcements**: GitHub Discussions + Twitter
- **Community chat**: Discord (announcement channel)
- **Maintainer sync**: Weekly video calls (Discord)
- **Email**: conduct@vitte-lang.dev (Code of Conduct issues)

### Response Times

- **Critical bugs**: 24 hours
- **PR reviews**: 3 business days
- **Issue responses**: 1 week
- **Community questions**: Best effort

## Conflict Resolution

For disputes between maintainers:

1. **Discussion** - Talk through the issue
2. **Mediation** - Project lead facilitates
3. **Vote** - Simple majority decides if needed
4. **Appeal** - Can request steering committee review
