[@_admin] La CLI, expliquée simplement
▲ 139 • cli
La CLI de Vitte n'est pas là pour faire joli: elle sert à répondre vite à trois questions très humaines. Est-ce que ça compile ? Si ça casse, pourquoi ? Et si je veux aller plus loin, quelle commande me donne la prochaine marche ?
Dans la pratique, le trio utile reste simple: vitte explain pour comprendre un diagnostic, vitte check pour valider un morceau de code, et les cibles Make quand on veut lancer quelque chose de plus large sans retenir une dizaine d'options.
Le but n'est pas de pousser l'utilisateur à apprendre des commandes par cœur. Le but, c'est que chaque commande réponde à un besoin clair et qu'elle rende la suite du travail plus lisible. Quand la sortie est nette, on perd moins de temps à deviner ce qui se passe.
C'est aussi pour ça que la documentation de la CLI reste proche des diagnostics et du book: on lit un message, on comprend quoi corriger, puis on revient à la commande qui donne le bon contexte. C'est une petite boucle, mais elle évite beaucoup de bruit.

[@_vincent.rs] Stdlib, source, et contrat de lecture
stdlib
La stdlib n'est pas juste un dossier de plus: c'est la surface que les gens touchent le plus souvent quand ils veulent écrire quelque chose de concret, le brancher à un projet, ou comprendre pourquoi un import est accepté d'un côté et refusé de l'autre.
Dans le dépôt, on garde le chemin source réel comme repère technique, mais dans la documentation on parle surtout de la bibliothèque stdlib, du contrat qui l'encadre, et des règles qui évitent que les exemples deviennent flous.
Concrètement, le travail récent a surtout servi à trois choses: clarifier le vocabulaire, relier les pages du site aux chapitres utiles, et faire en sorte que les diagnostics pointent vers une explication simple au lieu d'un message sec.
Ce qui compte ici, ce n'est pas d'empiler des termes techniques, mais de garder une ligne lisible: une stdlib qui se comprend vite, des imports qui se prédisent, et une documentation qui raconte le chemin sans se contredire.

[@_roussov] stdlib/kernel: syntaxe nouvelle, exemple complet
▲ 149 • kernel
Je voulais montrer un vrai morceau de code Vitte: court, lisible, et utile pour relire la nouvelle syntaxe sans chercher le contexte ailleurs.
use stdlib/kernel as kernel
pull stdlib/kernel/io as io
pull stdlib/kernel/math as math

proc entry main() -> Int {
  let title = kernel::form("kernel demo")
  let value = math::clamp(128, 0, 100)

  when value {
    100 -> io::give("kernel ready: " + title)
    otherwise -> io::give("kernel fallback: " + title)
  }

  give value
}
Lecture rapide: use pose le noyau, pull ajoute les sous-modules, puis when / otherwise gardent le flux visible. Le mot kernel donne l'ancre, et la fonction reste assez petite pour être relue d'un coup.
En pratique, ce genre d'exemple fait le lien entre la stdlib, le contrat qui l'encadre, et les diagnostics qui pointent vers une correction concrète.

[@_roussov] kernel: cas d'erreur et diagnostic attendu
▲ 92 • diagnostic
Même base, mais cette fois je montre l'erreur la plus utile à garder en tête: quand l'import n'est plus aligné avec le contrat de la stdlib, il faut un message qui dit quoi corriger, pas juste que quelque chose a cassé.
use stdlib/kernel as kernel
pull stdlib/kernel/io as io

proc entry main() -> Int {
  give io::missing_entry()
}
Diagnostic visé: le cas doit remonter vers un message de type E1016 quand le point d'entrée interne attendu n'est pas disponible, ou vers un diagnostic d'import si le chemin n'est pas canonique. Le but n'est pas de cacher la panne, mais de dire immédiatement où regarder.
Ce qu'on veut lire: un message qui nomme la bibliothèque, le symbole attendu, et l'action minimale à faire ensuite. C'est ce qui rend la stdlib agréable à utiliser au quotidien: le bug est visible, mais la correction reste simple.
