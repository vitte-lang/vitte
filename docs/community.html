@roussov on the Vitte compiler

@roussov wrote this note for the community thread. The GitHub path https://github.com/vitte-lang/vitte/tree/main/src/vitte/compiler does not resolve as written, so the stable place to follow the compiler surface is Source, with the driver notes in Compiler Driver Migration.

This page is a working note. What matters in a compiler is not the amount of machinery around it, but the discipline of the layers. A compiler should read like a sequence of decisions that can be named, tested, and replaced one by one. The front of the pipeline should stay small enough to explain, the middle should stay explicit enough to inspect, and the back end should stay honest about what it consumes and what it emits. Keeping the compiler in Vitte means the code that defines the language, the code that checks it, and the code that lowers it all live in the same vocabulary.

The compiler surface is easiest to maintain when each step answers a narrow question. Is this token valid? Does this form belong in the grammar? Can this declaration be reduced to a stable internal shape? Is the diagnostic precise enough to help someone move forward? Does the backend preserve the promise made by the source syntax? When those questions are separated, the implementation becomes calmer, and the documentation becomes easier to trust, because every page can point to one layer instead of pretending the whole system is a single black box.

That is why the current Vitte compiler story is worth telling carefully. The driver is no longer just a wrapper around a separate host implementation, the grammar is not treated as a side file, and the diagnostics are not an afterthought. The compiler now behaves like a Vitte part of the repository. That means the community can talk about it in ordinary terms: source surface, parser surface, IR surface, backend surface, and release surface. It also means that the maintenance work is visible. If a rule changes, the change should show up in the book, in the diagnostics, in the driver, and in the tests without needing a translation layer between languages.

@roussov, the latest compiler pass pushed the code closer to that shape in a very concrete way: src/vitte/compiler/driver/pipeline.vit was reduced to orchestration and artifact assembly, while the real parsing and lowering logic moved into src/vitte/compiler/ir/ast.vit with module-level parsing, top-level routing, proc signatures, statement parsing, and direct AST-to-HIR/MIR conversion now living beside the IR types instead of inside the driver; that gives us a cleaner split between source parsing, IR construction, and emission, and it also tightened the diagnostics so missing or invalid space and proc declarations are reported from the parse path itself. The smoke coverage in src/vitte/compiler/tests/smoke.vit was kept aligned with that change, so the nominal path and the statement lowering checks now exercise the real IR surface instead of legacy driver mirrors, which makes the current compiler progress easier to trust, easier to extend, and much easier to explain to contributors who want to see exactly where let, set, and give land in the pipeline.

@goginx

The latest changes do not make the compiler larger, they make it clearer: the driver now only gathers what the IR really needs, and the actual logic lives in src/vitte/compiler/ir/ast.vit.

Module parsing moved to the place where the language is described, so space, use, proc, let, set, and give are no longer duplicated in the driver.

Diagnostics are now structured: missing space, malformed proc signatures, and broken statements are carried as typed data before the frontend formats them for display.

That matters because a compiler without clear diagnostics is only half finished; the new shape makes it possible to track an error's origin, line, and kind separately.

The frontend was aligned too: src/vitte/compiler/frontend/diagnostics.vit is now the shared place for formatting, counting, and span data.

The smoke tests therefore no longer rely on local special forms; they use the same diagnostic type that the compiler driver sees.

In parallel, stdlib compression was organized: compression.vitl now binds deflate, brotli, algorithms, lz, huffman, and stats as real submodules.

The old mixed forms with legacy syntax were replaced so the modules can speak to the current parser again and provide clean manifests, health checks, and selftests.

The shared compression smoke test now exercises the whole family in one pass and shows that the new module structure is not only documented, but actually executed.

That makes the direction pretty clear: fewer hidden special cases, fewer duplicate paths, and a more direct link between code, diagnostics, and documentation.

If you look at the older parts of the tree, the basic idea stays the same: calm the structure first, clean up the surface second, then bind the tests to that shape.

That is why the latest changes are not just a batch of fixes, but a real step toward a compiler that treats its own layers seriously.

For contributors, that is a good contract. If you want to improve parsing, you should be able to read the parser chapter and the grammar files without hunting through a foreign implementation. If you want to change a diagnostic, you should see the rule that produces it and the place where it belongs in the compiler flow. If you want to adjust the command line, you should compare the driver, the completion snapshots, and the release notes. That kind of alignment is what makes a compiler feel real rather than ceremonial.

The larger rule is simple: keep the compiler readable enough that a newcomer can trace one feature from syntax to output. Keep the source of truth in Vitte whenever possible. Keep the docs in step with the code, not behind it. Keep the terminology stable so that community discussions can refer to the same objects as the implementation. A compiler becomes easier to evolve when it is written as a set of honest, checkable transitions instead of a monolith with hidden edges.

So if you are following the Vitte compiler from the outside, start with the source view and then move inward: the driver, the grammar, the tests, the IR, the backend, and the diagnostics. If you are following it from the inside, do the reverse and make sure each layer still says the same thing. That is the kind of discipline that lets a project grow without losing its shape, and it is the reason the compiler can now be discussed as a first-class Vitte artifact rather than a borrowed implementation detail.

In short, the compiler should remain easy to explain, easy to inspect, and easy to evolve. That is the kind of message the community can build on, and it is the kind of structure that keeps the work useful as the language grows.

Compiler principles

  • One layer, one responsibility. The parser should not guess about backend policy, and the backend should not carry syntax concerns.
  • One rule, one place. If a constraint exists, it should be written once in the grammar, once in the driver, or once in the diagnostics, not scattered across all three.
  • One example, one expectation. A documented example should compile for a reason that can be named, not because the pipeline happens to accept it by accident.
  • One message, one action. Diagnostics should tell the user what failed, where it failed, and what to try next.
  • One source of truth. When Vitte owns the compiler code, the docs, tests, and release pages can stay synchronized without translation work.

This is also why the compiler discussion belongs in public. The community should not need to infer structure from fragments. It should be able to read the project as a sequence of contracts: syntax contracts, semantic contracts, lowering contracts, backend contracts, and release contracts. Once those contracts are visible, contributors can work in parallel without stepping on each other, because the shape of the system is already written down.

If you only want the short version, see News. If you want to inspect the implementation path, stay on Source and move outward from there.
