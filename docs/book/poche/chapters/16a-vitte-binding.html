Extended Example

// Scenario liaison Vitte: execution complete et verifiable
space demo/liaison-vitte

form AbiEnvelope { version: int payload_size: int flags: int }
pick VitteCall { case Ok(code: int) case Err(code: int) }

proc abi_version() -> int { give 3 }

// Validation ABI: refuse toute incompatibilité avant appel natif
proc validate_abi(e: AbiEnvelope) -> int {

  if e.version != abi_version() { give 51 }

  if e.payload_size <= 0 { give 52 }

  if e.payload_size > 4096 { give 53 }

  if e.flags < 0 { give 54 }

  give 0
}

// Appel natif simulé: exécution seulement si la règle est valide
proc call_vitte(e: AbiEnvelope) -> VitteCall {
  let v: int = validate_abi(e)

  if v != 0 { give VitteCall.Err(v) }

  if e.payload_size % 2 == 0 { give VitteCall.Ok(0) }

  give VitteCall.Err(55)
}

// Conversion finale vers un code de sortie
proc to_exit(r: VitteCall) -> int {

  match r {
    case Ok(c) { give c }
    case Err(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let e: AbiEnvelope = AbiEnvelope(3, 128, 1)
  let r: VitteCall = call_vitte(e)

  return to_exit(r)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • abi_version: reads aucun paramètre and returns int.
  • validate_abi: reads e: AbiEnvelope and returns int.
  • call_vitte: reads e: AbiEnvelope and returns VitteCall.
  • to_exit: reads r: VitteCall and returns int.

2. Actual execution order

  1. The program enters main.
  2. call_vitte is called to process the next step.
  3. to_exit is called to process the next step.
  4. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: e.version != abi_version().
  • Test evaluated: e.payload_size <= 0.
  • Test evaluated: e.payload_size > 4096.
  • Test evaluated: e.flags < 0.
  • Test evaluated: v != 0.
  • Test evaluated: e.payload_size % 2 == 0.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> call_vitte -> to_exit -> sortie 0.
  • Example error: entrée invalide -> call_vitte renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Valid ABI -> output 0.
  • Incompatible version -> exit 51.
  • Payload out of order -> exit 52 or 53.

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
