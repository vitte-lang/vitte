37. Large-scale module conventions (pocket version)

Level: Advanced

Prerequisites: docs/book/chapters/09-modules.html. See also: docs/book/chapters/32-modules-catalog.html.

Concrete Problem

Real situation: Large-scale module conventions are best understood by replaying the program as an executable algorithm. You read the incoming data, the condition evaluated, and then the returned value. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: move from passive reading to proof: same input, same branch, same expected output.

Goal

Structuring a large code base with space, pull, use, share without excessive coupling.

Recommended Conventions

  1. space aligns with the domain (ex: core/auth, core/http).
  2. pull reserved for clear internal dependencies.
  3. use for the API consumed locally.
  4. share limits to stable symbols.

Minimal example

space core/auth
pull core/crypto as crypto
use core/user.{UserRepo}
share login, logout

Scaling rules

  • Prohibit circular dependencies between spaces.
  • Limit module path depth.
  • Stabilize public entry points.

Checklist

  1. Does the module have a unique responsibility?
  2. Are imports minimal?
  3. Are share symbols versioned?

Progressive examples (N1 -> N3)

N1 (base): minimum module

space core/auth
proc login() -> int { give 0 }
share login

N2 (intermediate): explicit imports

space core/auth
pull core/crypto as crypto
use core/user.{UserRepo}
proc login() -> int { give 0 }
share login

N3 (advance): stable public surface

space core/auth
proc login() -> int { give 0 }
proc logout() -> int { give 0 }
share login, logout

Anti-example

space core/auth
proc internal_only() -> int { give 0 }
share all

Quick validation

  1. Check border share.
  2. Avoid module cycles.
  3. Limit imports to what is necessary.

Why it matters

This section explains the practical value: reducing errors, accelerating diagnosis and stabilizing developments.

Quick check

Control question: If you change a key assumption, what outcome should change and why?

Try this

  1. Run the nominal example.
  2. Introduce an error case.
  3. Check the output and document the deviation.

If you're stuck

Fixed: keep the simplest version that respects the rule, then add a non-regression test.

Extended Example

// Scenario conventions modules echelle: execution complete et verifiable
space demo/conventions-modules-echelle

form ModuleSpec { name_len: int major: int minor: int }
pick Resolve { case Loaded(path_code: int) case Missing(code: int) }

proc normalize_len(n: int) -> int {

  if n <= 0 { give 0 }

  give n
}

// Résolution: valide la référence et produit une décision de chargement
proc resolve(m: ModuleSpec) -> Resolve {
  let n: int = normalize_len(m.name_len)

  if n == 0 { give Resolve.Missing(71) }

  if m.major <= 0 { give Resolve.Missing(72) }

  if m.minor < 0 { give Resolve.Missing(73) }
  let code: int = 100 + (m.major * 10) + m.minor

  give Resolve.Loaded(code)
}

// Conversion finale vers un code de sortie
proc to_exit(r: Resolve) -> int {

  match r {
    case Loaded(_) { give 0 }
    case Missing(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let m: ModuleSpec = ModuleSpec(8, 1, 2)
  let r: Resolve = resolve(m)

  return to_exit(r)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • normalize_len: reads n: int and returns int.
  • resolve: reads m: ModuleSpec and returns Resolve.
  • to_exit: reads r: Resolve and returns int.

2. Actual execution order

  1. The program enters main.
  2. resolve is called to process the next step.
  3. to_exit is called to process the next step.
  4. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: n <= 0.
  • Test evaluated: n == 0.
  • Test evaluated: m.major <= 0.
  • Test evaluated: m.minor < 0.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> resolve -> to_exit -> sortie 0.
  • Example error: entrée invalide -> resolve renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Valid specification -> output 0.
  • Invalid name -> exit 71.
  • Invalid version -> output 72 or 73.

7. Line by line (variables + values)

Convenient reading: Follow the variables in the actual order of execution, then check the observed output.

  • Entry point:
  • entry main at core/app launches the complete scenario.
  • Block functions:
  • normalize_len reads n: int then returns int.
  • resolve reads m: ModuleSpec then returns Resolve.
  • to_exit reads r: Resolve then returns int.
  • Variables created (initial value):
  • n: int starts with normalize_len(m.name_len).
  • code: int starts with 100 + (m.major * 10) + m.minor.
  • m: ModuleSpec starts with ModuleSpec(8, 1, 2).
  • r: Resolve starts with resolve(m).
  • Conditions that change the path:
  • if n <= 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if n == 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if m.major <= 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if m.minor < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • Nominal trace (example values):
  • initialization: n=normalize_len(m.name_len) -> code=100 + (m.major * 10) + m.minor -> m=ModuleSpec(8, 1, 2) -> r=resolve(m)
  • sequence: resolve -> to_exit
  • final output on this path: to_exit(r).
  • Error trace (example values):
  • if n <= 0 becomes true, the function returns 0.
  • Quick check:
  • restart with normal input and note the output,
  • restart with an invalid entry and note the error code,
  • confirm that the same input always produces the same output.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Rapid Diagnosis

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
