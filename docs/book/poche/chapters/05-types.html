5.0.1 When to avoid int

Avoid int when the size must remain stable between machines:

  • Binary API and Vitte binding.
  • network protocol and file format.
  • persistent serialization.
  • hardware registers and kernel/bare-metal code.

In these cases, impose an explicit width (i32/i64/i128/u32/u64/u128) and keep this width end-to-end.

Reference: see the Vitte Glossary in docs/book/glossary.html and the Editorial Checklist in docs/book/checklist.html. Complement: docs/book/classic-mistakes.html.

5.1 Describe data with a structural type

A structural type groups together several named fields that form a single logical entity. It is no longer a question of "any two integers", but of a point in the plan with an explicit business meaning.

form Point {
  x: int
  y: int
}
proc manhattan(p: Point) -> int {
  give p.x + p.y
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. No critical branching here: the flow is sequential, line by line.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns p.x + p.y.
  5. Beginner concept: a variable is a named memory box which contains a value of a declared type.
  6. Beginner concept: a Boolean condition is worth vrai or faux; it is this which determines the branch executed.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

Point expresses a clear idea: a value with two coordinates x and y. The manhattan procedure therefore does not accept just any data; it requires a Point. This constraint is valuable because it prevents ambiguous calls and makes the intent obvious in the signature.

At runtime, the behavior remains simple:

  • with Point(3,4), the function returns 7.
  • with Point(10,-2), it returns 8.

But the real value of the example is not the calculation itself. It is in the readability of the rule: by seeing manhattan(p: Point), the reader immediately understands the nature of the expected data.

Reflex to keep: when data has a business meaning, give it a business type. You thus avoid the implicit exchanges of "two integers which look alike" but do not mean the same thing.

Classic mistakes to avoid:

  • handle separate tuples or integers instead of an explicit business type.
  • give field names that are too vague, which hide the intention.
  • multiply unnecessary conversions around an already well-defined type.

5.2 Representing alternatives with a sum type

Some values ​​do not have a single possible form. A result can succeed, or it can fail. A parser can produce a value, or an error. The sum type is used precisely to encode these alternatives without ambiguity.

pick Result {
  case Ok(value: int)
  case Err(code: int)
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. No critical branching here: the flow is sequential, line by line.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: This block prepares a state, the output is calculated further in the pipeline.
  5. Beginner concept: a variable is a named memory box which contains a value of a declared type.
  6. Beginner concept: a Boolean condition is worth vrai or faux; it is this which determines the branch executed.

Quick check: Trace an execution with concrete values ​​(x=..., i=...) to confirm the branch actually taken.

Here, Result describes a closed universe of valid states: either Ok with an integer value, or Err with an error code. This point is essential. The program no longer has to guess the real state of a piece of data based on implicit conventions; the state is named and carried by the type.

At runtime, any value of type Result is necessarily one of the two variants. You can't "invent" an unintended third state without changing the type itself. This closure reduces impossible states and simplifies maintenance: when behavior changes, you know exactly which forms of data are affected.

In practice, this model also improves error messages and testing:

  • tests target explicit variants.
  • business errors remain named.
  • unmanaged cases become visible sooner.

Classic mistakes to avoid:

  • use a "sentinel" integer (e.g. -1) in place of an error variant.
  • mix data of different types in the same field.
  • introduce implicit cases that the type does not document.

5.3 Processing a variant with match

Once the sum type is defined, the correct reading consists of explicitly processing each variant. The match aligns flow control with the type structure.

proc unwrap_or_zero(r: Result) -> int {

  match r {
    case Ok(v) { give v }
    case Err(_) { give 0 }
    otherwise { give 0 }
  }
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Branch selection: match r chooses a case according to the current state.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns v.
  5. Beginner concept: a variable is a named memory box which contains a value of a declared type.
  6. Beginner concept: a Boolean condition is worth vrai or faux; it is this which determines the branch executed.

Quick check: Trace an execution with concrete values ​​(x=..., i=...) to confirm the branch actually taken.

The unwrap_or_zero function expresses a concrete rule: if the result is Ok, we recover its value; if it is an error, we return 0. The flow follows variations of Result exactly, making the behavior immediate to verify.

Step by step reading:

  • if r = Ok(42), the first branch matches, return 42.
  • if r = Err(500), the second branch matches, return 0.
  • the otherwise branch serves as a safety net here.

The code doesn't "guess" anything: it inspects the shape of the value, then applies the associated rule. It is this correspondence between type and branches that makes the system robust.

Classic mistakes to avoid:

  • treat only one variant and forget the others.
  • write branches that contradict the business sense of the cases.
  • add a otherwise to hide a design oversight rather than correct the model.

Keep in mind

Typing becomes truly useful when it covers the business, not just the technical form of the data. A structural type clarifies what an entity is. A sum type clarifies the possible states of this entity. A well-written match clarifies how one reacts to each of these states. With these three tools, your code gains in readability, security and stability: it becomes more difficult to express an error, and easier to explain an intention.

Practical quality criterion for this chapter:

  • you can say what each type represents in one clear sentence.
  • you know how to list all the possible variations of a result.
  • you can reread a match and check that no important cases are missed.

Extended Example

// Scenario types: execution complete et verifiable
space demo/types

form Input { id: int value: int quota: int }
pick Eval { case Accepted(score: int) case Rejected(code: int) }

proc validate(x: Input) -> Eval {

  if x.id <= 0 { give Eval.Rejected(21) }

  if x.quota < 0 { give Eval.Rejected(22) }

  if x.value < 0 { give Eval.Rejected(23) }

  give Eval.Accepted(x.value)
}

proc transform(score: int, quota: int) -> int {
  let capped: int = score
  if capped > quota { set capped = quota }

  if capped < 0 { give 0 }

  give capped * 2
}

proc decide(r: Eval, quota: int) -> Eval {

  match r {
    case Accepted(s) {
      let out: int = transform(s, quota)

      if out >= 10 { give Eval.Accepted(out) }

      give Eval.Rejected(31)
    }
    case Rejected(c) { give Eval.Rejected(c) }
    otherwise { give Eval.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(r: Eval) -> int {

  match r {
    case Accepted(_) { give 0 }
    case Rejected(code) { give code }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let x: Input = Input(1, 8, 9)
  let v: Eval = validate(x)
  let d: Eval = decide(v, x.quota)

  return to_exit(d)
}

Detailed explanation of the big block

You read this big block as a complete scenario: data preparation, processing, then final output.

1. Role of each party

  • Starting point: entry main at core/app.
  • validate: reads x: Input and returns Eval.
  • transform: reads score: int, quota: int and returns int.
  • decide: reads r: Eval, quota: int and returns Eval.
  • to_exit: reads r: Eval and returns int.

2. Actual execution order

  1. The program enters main.
  2. validate is called to process the next step.
  3. decide is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: x.id <= 0.
  • Test evaluated: x.quota < 0.
  • Test evaluated: x.value < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 0.
  • Test evaluated: out >= 10.
  • Selection by match r: the path depends on the state received.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> validate -> decide -> to_exit -> sortie 0.
  • Example error: entrée invalide -> validate renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal case -> output 0.
  • Strict quota case -> stable behavior.
  • Invalid case id<=0 -> output 21.

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.
