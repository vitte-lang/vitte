29. Architecture Style (pocket version)

Level: Advanced

Prerequisites: previous chapter docs/book/chapters/28-conventions.html and docs/book/glossary.html. See also: docs/book/chapters/28-conventions.html, docs/book/chapters/30-faq.html, docs/book/glossary.html.

Concrete Problem

Real situation: Architecture style becomes clear when we trace each step of the calculation. The goal is to connect line of code and concrete effect on the output. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: move from passive reading to proof: same input, same branch, same expected output.

Why it matters

This chapter gives you a clear understanding of Architecture Style. You will find the framework, the invariants and the reading decisions useful in practice.

What you will do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

Recommended first reflex: read the entries and conditions first, then only the syntactic form.

Step-by-step explanation

  1. Identify the intent of the block.
  2. Check the main condition or test.
  3. Confirm the observable output.
  4. Note how this block serves Architecture Style throughout the chapter.

Common pitfalls

  • Read the syntax without checking the behavior.
  • Mix general rule and error cases in the same explanation.
  • Introduce an optimization before having stabilized the flow of Architecture style.

Short exercise

Take an example from the chapter on Architectural Style. Change a condition or input value, then check if the result remains as expected.

Summary in 5 points

  1. You know the objective of the chapter on Architectural Style.
  2. You know how to read an example of the chapter in a structured way.
  3. You distinguish nominal case and error case.
  4. You avoid the most common pitfalls.
  5. You can reuse these rules in the next chapter.

29.1 Write a pure domain without infrastructure dependencies

space app/domain
form Order {
  amount: int
}
proc approve(o: Order) -> bool {
  give o.amount > 0
}

Reading line by line (beginner):

  1. space app/domain: This line marks a clear step in the flow.
  2. form Order {: this line opens the Order structure which groups coherent data under the same business name, useful for keeping a stable vocabulary.
  3. amount: int: this line declares the field amount with the type int, which documents its role and limits handling errors.
  4. }: this brace closes the block.
  5. proc approve(o: Order) -> bool {: the rule is defined for approve: inputs o: Order and output bool, it clarifies the intention before detailed reading of the body.
  6. give o.amount > 0: the branch returns o.amount > 0 for the current branch, the output is explicit and easy to verify.
  7. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: without blocking test, the main branch returns o.amount > 0.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should trigger a clear validation or fallback path.

The intent of this step is straightforward: to isolate a pure, testable domain with no infrastructure dependencies.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, approve(Order(10))=true and approve(Order(0))=false.

Common mistakes to avoid:

  • encoding implicit conventions instead of carrying them by type.
  • mix different business cases in the same representation.
  • add variants without updating treatment points.

29.2 Translate the business into a service layer

space app/service
pull app/domain as d
proc handle(amount: int) -> int {
  let ok: bool = d.approve(d.Order(amount))

  if ok { give 200 }

  give 422
}

Reading line by line (beginner):

  1. space app/service: This line marks a clear step in the flow.
  2. pull app/domain as d: This line marks a clear step in the flow.
  3. proc handle(amount: int) -> int {: the rule is set for handle: inputs amount: int and output int, it clarifies the intention before detailed reading of the body.
  4. let ok: bool = d.approve(d.Order(amount)): this line creates the variable ok of type bool to explicitly name an intermediate step of the reasoning.
  5. if ok { give 200 }: this test handles the invalid case before the main path.
  6. give 422: the branch returns 422 for the current branch, the output is explicit and easy to verify.
  7. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: if ok is true, the output becomes 200.
  • Nominal case: without blocking test, the main branch returns 422.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should trigger a clear validation or fallback path.

The intention of this step is direct: to make the service play a translation role between business and application output.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, handle(10)=200 and handle(0)=422.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

29.3 Keeping a thin entry point into the IO layer

space app/io
pull app/service as s

// Point d'entree du scenario
entry main at core/app {
  let code: int = s.handle(10)

  return code
}

Reading line by line (beginner):

  1. space app/io: This line marks a clear step in the flow.
  2. pull app/service as s: This line marks a clear step in the flow.
  3. entry main at core/app {: This line sets the entry point main into core/app and serves as an end-to-end executable scenario for the chapter.
  4. let code: int = s.handle(10): this line creates the variable code of type int to explicitly name an intermediate step of the reasoning.
  5. return code: this line ends the execution of the current block with the code code, useful for observing the overall result of the scenario.
  6. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the main scenario ends with return code.
  • Testable observation: Running the scenario verifies the exit code code.

Quick check: invalid input should trigger a clear validation or fallback path.

The intention of this step is straightforward: keep a thin entry point, limited to orchestration.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, the scenario shows a final return 200.

  • accumulating special cases without clarifying intent.

  • leave implicit decisions that slow down rereading.

Keep in mind

Pure domain, translation service and minimal entry form a stable long-term architecture. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

Practical quality criterion for this chapter:

  • you know how to say in which layer a business rule should live.
  • you can modify the IO without touching the domain.
  • you can test the main logic without starting the full application.

Quick check

Question: What happens if the entry is invalid? Expected response: An explicit test or stable fallback path should apply.

Try this

  1. Take an example from the chapter and modify a test condition to observe different behavior.
  2. Write a mini mental test on an invalid entry in the chapter, then predict the executed branch.

If you're stuck

  • identify the modified line and explain in one sentence the new expected output.
  • Name the test or fallback branch actually used.

Extended Example

// Scenario style: execution complete et verifiable
space demo/style

form Input { id: int value: int quota: int }
pick Eval { case Accepted(score: int) case Rejected(code: int) }

proc validate(x: Input) -> Eval {

  if x.id <= 0 { give Eval.Rejected(21) }

  if x.quota < 0 { give Eval.Rejected(22) }

  if x.value < 0 { give Eval.Rejected(23) }

  give Eval.Accepted(x.value)
}

proc transform(score: int, quota: int) -> int {
  let capped: int = score
  if capped > quota { set capped = quota }

  if capped < 0 { give 0 }

  give capped * 2
}

proc decide(r: Eval, quota: int) -> Eval {

  match r {
    case Accepted(s) {
      let out: int = transform(s, quota)

      if out >= 10 { give Eval.Accepted(out) }

      give Eval.Rejected(31)
    }
    case Rejected(c) { give Eval.Rejected(c) }
    otherwise { give Eval.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(r: Eval) -> int {

  match r {
    case Accepted(_) { give 0 }
    case Rejected(code) { give code }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let x: Input = Input(1, 8, 9)
  let v: Eval = validate(x)
  let d: Eval = decide(v, x.quota)

  return to_exit(d)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.
  • validate: reads x: Input and returns Eval.
  • transform: reads score: int, quota: int and returns int.
  • decide: reads r: Eval, quota: int and returns Eval.
  • to_exit: reads r: Eval and returns int.

2. Actual execution order

  1. The program enters main.
  2. validate is called to process the next step.
  3. decide is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: x.id <= 0.
  • Test evaluated: x.quota < 0.
  • Test evaluated: x.value < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 0.
  • Test evaluated: out >= 10.
  • Selection by match r: the path depends on the state received.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> validate -> decide -> to_exit -> sortie 0.
  • Example error: entrée invalide -> validate renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal case -> output 0.
  • Strict quota case -> stable behavior.
  • Invalid case id<=0 -> output 21.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
