69. Complete CLI production-ready project (pocket version)

Level: Intermediate

Prerequisites: docs/book/chapters/21-cli-project.html, docs/book/chapters/58-ci-pipeline-for-vitte-project.html. See also: to be defined.

Concrete Problem

Real situation: for production-ready CLI Complete Project, the question is not 'which rule to write' but 'which path the code really takes'. This reading by execution avoids vague interpretations. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: you find the same pipeline to observe what really changes when you modify a branch. Educational objective: Reread a block, predict the output, then confirm the prediction with a simple, repeatable execution.

Goal

Build a robust CLI with packaging, tests, diagnostics and CI.

Deliverables

  1. Main commands and help.
  2. Clear user error handling.
  3. Unit testing + integration.
  4. Green CI pipeline.

Progressive examples (N1 -> N3)

N1 (basic): nominal CLI command

Snippet Vitte:

entry main at app/cli {
  return 0
}

Order:

make grammar-check

N2 (intermediate): user error checked

Snippet Vitte:

proc parse_arg(ok: bool) -> int {

  if not ok { give 2 }

  give 0
}

Order:

make grammar-test

L3 (advanced): full CI execution

Snippet Vitte:

proc run_cli(ok: bool) -> int {

  if not ok { give 2 }

  give 0
}
entry main at app/cli { return run_cli(true) }

Commands:

make grammar-gate
make book-qa

Anti-example

CLI sans code de sortie stable, sans tests de cas d'erreur, sans validation CI.

Quick validation

  1. Check nominal/error output codes.
  2. Check green CI commands.
  3. Check minimal user documentation.

Why it matters

This section explains the practical value: reducing errors, accelerating diagnosis and stabilizing developments.

Quick check

Control question: If you change a key assumption, what outcome should change and why?

Try this

  1. Run the nominal example.
  2. Introduce an error case.
  3. Check the output and document the deviation.

If you're stuck

Fixed: keep the simplest version that respects the rule, then add a non-regression test.

Extended Example

// Scenario projet cli production ready: execution complete et verifiable
space demo/projet-cli-production-ready

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

Detailed explanation of the big block

You read this big block as a complete scenario: data preparation, processing, then final output.

1. Role of each party

  • Starting point: entry main at core/app.
  • parse_request: reads r: Request and returns Result.
  • apply_policy: reads total: int, quota: int and returns Result.
  • persist_sim: reads x: Result and returns Result.
  • to_exit: reads x: Result and returns int.

2. Actual execution order

  1. The program enters main.
  2. parse_request is called to process the next step.
  3. apply_policy is called to process the next step.
  4. persist_sim is called to process the next step.
  5. to_exit is called to process the next step.
  6. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: r.id <= 0.
  • Test evaluated: r.quota < 0.
  • Test evaluated: r.amount < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 5.
  • Test evaluated: v % 13 == 0.
  • Selection by match x: the path depends on the state received.
  • Selection by match x: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> parse_request -> apply_policy -> persist_sim -> to_exit -> sortie 0.
  • Example error: entrée invalide -> parse_request renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal request -> output 0.
  • Invalid input id<=0 -> output 91.
  • Business refusal value<5 -> exit 94.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Mini Case Study (Before / After)

Before: mixed business logic and technical output, expensive diagnosis. After: entrance tests, business decision, final screening separated; more readable and testable behavior. Impact: faster review, easier to locate regression.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
