23. System Project (pocket version)

Level: Advanced

Prerequisites: previous chapter docs/book/chapters/22-http-project.html and docs/book/glossary.html. See also: docs/book/chapters/22-http-project.html, docs/book/chapters/24-kv-project.html, docs/book/glossary.html.

Concrete Problem

Real situation: System Project is best understood by replaying the program as an executable algorithm. You read the incoming data, the condition evaluated, and then the returned value. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: each section uses the same scenario to isolate a single technical decision at a time. Educational objective: understand why a line exists and what it changes in the trajectory of the program.

Why it matters

This chapter gives you a clear understanding of System Project. You will find the framework, the invariants and the reading decisions useful in practice.

What you will do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

This chapter pursues a clear objective: to build a Vitte system core which remains auditable under real constraints. We will go beyond the simple memory example to pose a complete architecture: region model, bounded access, minimum unsafe points, composition of operations and test plan.

The challenge is not to “make” a machine instruction work, but to maintain readable invariants throughout the flow. In system code, readability is not a luxury: it is a security measure.

23.3 Implement bounded read and write

pub proc read_cell(m: SysMem, addr: int) -> int {

  if not contains(m.region, addr) { give 0 }
  let i: int = to_index(m.region, addr)

  give m.cells[i]
}
pub proc write_cell(m: SysMem, addr: int, v: int) -> SysResult {

  if not contains(m.region, addr) { give ErrBounds }
  let i: int = to_index(m.region, addr)
  m.cells[i] = v

  give Ok
}

23.5 Compose an atomic memory + synchronization operation

pub proc write_then_pause(m: SysMem, addr: int, v: int) -> SysResult {
  let w: SysResult = write_cell(m, addr, v)

  match w {
    case Ok {
      cpu_pause()

      give Ok
    }
    case ErrBounds { give ErrBounds }
    otherwise { give ErrState }
  }
}

Minimum clearance to cover:

  2. Upper limit: address base + size => rejection.
  3. Nominal case: internal address => consistent write/read.
  4. cpu_halt_if(false): This line marks a clear step in the flow.

23.9 Minimum system rule

Minimum CLI entry:

  • --op (read or write).
  • --addr logical address.
  • --value mandatory if --op=write.
  • --trace-id optional for log correlation.

Rights required:

  • read-only for read.
  • minimal writing to the targeted region for write.
  • no global privileges if the region can be isolated.

Outputs and error codes:

  • 0: success.
  • 2: invalid entry.
  • 13: permission denied.
  • 44: resource not found.
  • 70: inconsistent internal state.

23.10 Security invariants

Non-negotiable invariants:

  • any input path is canonicalized before use (path canonicalisé).
  • no dynamic shell execution from input (no shell injection).
  • application of the principle of least privilege (droits minimaux).
  • all memory terminals are validated before index conversion.

23.11 Complete traceable scenario

Concrete pipeline:

  1. Raw input: --op=write --addr=103 --value=42 --trace-id=t-001.
  2. Validation: check presence of fields, address range and rights.
  3. Syscall/mutation: write the targeted cell, then sync.
  4. Mapping result: Ok -> code 0, ErrBounds -> code 2, ErrState -> code 70.

Expected output:

  • stable return code.
  • log correlated by trace-id.
  • no mutation if validation fails.

23.12 System integration tests

  1. Nominal.
  2. Entry: write valid in authorized region.
  3. Expected: code 0, immediately readable value, success log.
  1. Permission denied.
  2. Entry: write without sufficient rights.
  3. Expected: code 13, no memory mutation, log denied.
  1. Resource not found.
  2. Input: target missing or unresolved region.
  3. Expected: code 44, unchanged state, explicit diagnosis.

23.13 Portability box

What changes depending on OS:

  • permissions model and rights checking API.
  • forms Vitte calls for pause/sync.
  • system return code conventions and error messages.

What remains stable:

  • CLI entry rule.
  • bound and atomicity invariants.
  • business mapping SysResult -> code de sortie.

23.14 Box Observability

Minimum requirements:

  • structured logs (ts, level, trace_id, op, addr, result).
  • return code always consistent with the business result.
  • trace_id propagated from input to output.

Recommended log format:

  • ts=<unix_ms> trace_id=<id> op=<read|write> addr=<n> result=<Ok|Err..> code=<n>.

23.15 Costs

I/O:

  • dominant cost when the operation affects disk, device or IPC.

Memory:

  • Low local cost on the nominal path, but sensitive to unnecessary copies.

Blocking system calls:

  • risk of latency and chain domino effect.
  • impose explicit timeout and recovery strategy.

23.16 Classic system project errors

  • TOCTOU: check then use a resource without a consistent lock.
  • Path traversal: accept a non-canonicalized path.
  • Forgotten Timeouts: Let a call hang indefinitely.
  • Implied permissions: assume an unverified right.
  • Late mapping: convert too late to observable exit codes.

Keep in mind

A robust system core is built by readable layers: centralized preconditions, bounded access, minimal unsafe, stable composition. The lower level the code, the more highly explicit the rule must be.

Practical quality criterion for this chapter:

  • you can prove that no out-of-bound write is possible.
  • you can point each zone unsafe in seconds.
  • you know how to replay a complete system scenario in a stable manner.

Extended Example

// Scenario projet sys: execution complete et verifiable
space demo/projet-sys

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
pub proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
pub proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
pub proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
pub proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • parse_request: reads r: Request and returns Result.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
