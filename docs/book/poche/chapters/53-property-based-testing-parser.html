53. Property-based testing for the parser (pocket version)

Level: Advanced

Prerequisites: docs/book/chapters/27-grammar.html, docs/book/chapters/52-grammar-valid-invalid-tests.html. See also: to be defined.

Concrete Problem

Real situation: Property-based testing for the parser is best understood by replaying the program as an executable algorithm. You read the incoming data, the condition evaluated, and then the returned value. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: understand why a line exists and what it changes in the trajectory of the program.

Goal

Automatically find parser error cases via data generation.

Useful properties

  1. Deterministic parse on same input.
  2. Invalid inputs never crash the parser.
  3. AST invariants respected after parse.

Approach

  1. Define input generators.
  2. Encode the properties.
  3. Auto reducer to minimize counterexamples.

Progressive examples (N1 -> N3)

N1 (base): deterministic input

entry main at app/demo { return 0 }

N2 (intermediate): robust invalids

entry main at app/demo { return }

N3 (advance): AST properties

entry main at app/demo {
  return 1 + 2 * 3
}

Anti-example

# generation aleatoire sans verification de propriete

Quick validation

  1. Valid parse determinism.
  2. No crashes on invalids.
  3. AST properties verified.

Why it matters

This section explains the practical value: reducing errors, accelerating diagnosis and stabilizing developments.

Quick check

Control question: If you change a key assumption, what outcome should change and why?

Try this

  1. Run the nominal example.
  2. Introduce an error case.
  3. Check the output and document the deviation.

If you're stuck

Fixed: keep the simplest version that respects the rule, then add a non-regression test.

Extended Example

// Scenario property based testing parser: execution complete et verifiable
space demo/property-based-testing-parser

form SourceUnit { bytes: int lines: int tokens_hint: int }
pick ParseState { case Parsed(nodes: int) case Failed(code: int) }

// Scan: transforme l'entrée brute en signal exploitable
proc scan(u: SourceUnit) -> int {

  if u.bytes <= 0 { give 0 }

  if u.lines <= 0 { give 0 }

  give (u.tokens_hint + u.lines)
}

// Parse: construit un état syntaxique stable
proc parse(token_count: int) -> ParseState {

  if token_count == 0 { give ParseState.Failed(101) }

  if token_count < 4 { give ParseState.Failed(102) }

  give ParseState.Parsed(token_count)
}

proc validate_structure(nodes: int) -> int {

  if nodes <= 0 { give 201 }

  if nodes > 200000 { give 202 }

  give 0
}

// Conversion finale vers un code de sortie
proc to_exit(p: ParseState) -> int {

  match p {
    case Parsed(n) {
      let v: int = validate_structure(n)

      if v != 0 { give v }

      give 0
    }
    case Failed(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let u: SourceUnit = SourceUnit(120, 12, 18)
  let t: int = scan(u)
  let p: ParseState = parse(t)

  return to_exit(p)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • scan: reads u: SourceUnit and returns int.
  • parse: reads token_count: int and returns ParseState.
  • validate_structure: reads nodes: int and returns int.
  • to_exit: reads p: ParseState and returns int.

2. Actual execution order

  1. The program enters main.
  2. scan is called to process the next step.
  3. parse is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: u.bytes <= 0.
  • Test evaluated: u.lines <= 0.
  • Test evaluated: token_count == 0.
  • Test evaluated: token_count < 4.
  • Test evaluated: nodes <= 0.
  • Test evaluated: nodes > 200000.
  • Selection by match p: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> scan -> parse -> to_exit -> sortie 0.
  • Example error: entrée invalide -> scan renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Valid unit -> output 0.
  • Empty input (bytes=0) -> output 101.
  • Oversized structure -> exit 202.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
