This chapter pursues a field objective: to build a Vitte CLI which remains clean when real errors occur. The problem of a command line tool is not to produce a result in the nominal case. The real problem is to keep the logic readable when the user forgets an argument, chooses an unknown mode, or strings together invalid uses. The method followed here is strict: type the errors, segment the steps, then centralize the final projection to the system.

The approach adopted is deliberately literal: each example must be readable as a short demonstration, with a clear intention, an explicit execution path and a verifiable conclusion. This rhythm is that of a manual: understand, execute, then retain the useful invariant.

The method remains constant: set an intention, implement it in a compact form, then observe precisely what the program guarantees at runtime.

Reference: see the Vitte Glossary in docs/book/glossary.html and the Editorial Checklist in docs/book/checklist.html. Complement: docs/book/classic-mistakes.html.

  • Processing: validation, business logic, transformations.
  • Output: business result or status code.
  • Invariant: property that must remain true at each step.

21.4 Project a stable system policy

pub proc to_exit_code(r: CliResult) -> int {

  match r {
    case Ok(_) { give 0 }
    case Err(MissingArgs) { give 64 }
    case Err(UnknownMode(_)) { give 65 }
    otherwise { give 70 }
  }
}

Reading line by line (beginner):

  1. pub proc to_exit_code(r: CliResult) -> int {: the rule is set for to_exit_code: inputs r: CliResult and output int, it clarifies the intention before detailed reading of the body.
  2. match r {: this line starts a stable dispatch on r: a single branch will be chosen according to the form of the analyzed value.
  3. case Ok(_) { give 0 }: this case describes Ok(_) and explains the associated business decision, which reduces reading ambiguities.
  4. case Err(MissingArgs) { give 64 }: this case describes Err(MissingArgs) and explains the associated business decision, which reduces reading ambiguities.
  5. case Err(UnknownMode(_)) { give 65 }: this case describes Err(UnknownMode(_)) and explains the associated business decision, which reduces reading ambiguities.
  6. otherwise { give 70 }: This line marks the fallback path clearly.
  7. }: this brace closes the block.
  8. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the flow follows the main branch and produces a stable output.
  • Testable observation: forcing the case Ok(_) confirms the expected branch.

Quick check: invalid input should trigger a clear validation or fallback path.

The intention of this step is direct: to separate business semantics from exit code policy.

Conversion to exit codes is a system integration topic, not a business topic. Releasing it in a dedicated function gives you a clear boundary: you can evolve the internal semantics without breaking shell conventions.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, a success gives 0, a missing arity gives 64, an unknown mode gives 65. The otherwise guarantees a safety net with 70.

Common mistakes to avoid:

  • mix business rules and system projection rules.
  • rewrite the same exit code mappings in multiple functions.
  • leave error cases without an explicit exit code.

Keep in mind

A robust Vitte CLI is not one that contains a lot of code, but one that exposes a clear grammar of decisions. Entry validated at the border, business routed in pure functions, centralized system projection, and entry reduced to the role of orchestrator. With this discipline, complexity increases without destroying readability. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

Practical quality criterion for this chapter:

  • you know how to trace the complete flow of a CLI call in a few steps.
  • you know how to modify a validation rule without touching the routing.
  • you know how to evolve exit codes without touching the business logic.

Extended Example

// Scenario projet cli: execution complete et verifiable
space demo/projet-cli

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
pub proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
pub proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
pub proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
pub proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

Decision According to Constraint

  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.
