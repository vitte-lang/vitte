31. Build errors (pocket version)

Level: Advanced.

Prerequisites: previous chapter docs/book/chapters/30-faq.html and docs/book/grammar-surface.ebnf. See also: docs/book/chapters/30-faq.html, docs/book/chapters/16-interop.html, docs/book/chapters/23-system-project.html.

Concrete Problem

Real situation: Build errors are best understood by replaying the program as an executable algorithm. You read the incoming data, the condition evaluated, and then the returned value. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: move from passive reading to proof: same input, same branch, same expected output.

Why it matters

This chapter gives you a clear understanding of Build Errors. You will find the framework, the invariants and the reading decisions useful in practice.

What you will do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

Recommended first reflex: read the entries and conditions first, then only the syntactic form.

Step-by-step explanation

  1. Identify the intent of the block.
  2. Check the main condition or test.
  3. Confirm the observable output.
  4. Note how this block serves Build Errors throughout the chapter.

Common pitfalls

  • Read the syntax without checking the behavior.
  • Mix general rule and error cases in the same explanation.
  • Introduce an optimization before having stabilized the flow of Build errors.

Short exercise

Take an example from the chapter on Build Errors. Change a condition or input value, then check if the result remains as expected.

Summary in 5 points

  1. You know the objective of the chapter on Build Errors.
  2. You know how to read an example of the chapter in a structured way.
  3. You distinguish nominal case and error case.
  4. You avoid the most common pitfalls.
  5. You can reuse these rules in the next chapter.

Local level

  • Local level pipeline triage: Advanced.
  • Local syntactic correction level: Intermediate.
  • Local runtime/ABI level: Advanced.

This chapter serves as an operational guide to diagnosing a Vitte build failure in less than five minutes, from the first raw error to the validated patch.

Reading line by line (beginner):

  1. emit 1: statement placed at the top-level.
  2. # casse: statement top-level.: local diagnostic comment.

Fix:

  emit 1

  return 0
}

Reading line by line (beginner):

  1. entry main at app/repro {: opens a valid executable block at top-level.
  2. emit 1: statement now in an authorized block.
  3. return 0: Explicitly ends the scenario.
  4. }: cleanly closes the input block.
<<< minimal reproducer link >>> 
entry main at app/repro {
  return native_missing(1)
}
# parse OK, link KO: symbole absent.

Reading line by line (beginner):

  1. entry main at app/repro {: the top-level form is valid.
  2. return native_missing(1): call of an external symbol not provided.
  3. }: correct end of the block.

Fix: provide native_missing in runtime or override the call.

31.4 ABI/runtime checklist (common kernel/interop)

Common checklist:

  • Consistent ABI signatures between Vitte and Vitte.
  • runtime symbols exported and linked.
  • headers/runtime present (vitte_runtime.vitl, basic types).
  • freestanding mode respected (no implicit userland dependency).
  • Homogeneous calling conventions on all targets.

31.5 Glossary of common errors

  • E0007: unexpected top-level syntactic form.
  • E001x (mismatch type family): type rule violated.
  • undefined symbol: symbol missing when editing links.
  • invalid module path: malformed module/import path.

31.6 Playbook 5 minutes

  1. Read the first error and identify the layer.
  2. Reduce the file to the smallest reproducer.
  3. Correct a single local cause.
  4. Relaunch vitte build.
  5. Validate the doc with qa_book.py then qa_book.py --strict.

Extended Example

// Scenario erreurs build: execution complete et verifiable
space demo/erreurs-build

form Event { code: int severity: int payload_len: int }
pick Diagnostic { case Info(code: int) case Warn(code: int) case Error(code: int) }

// Classification: mappe un événement vers un niveau explicite
proc classify(e: Event) -> Diagnostic {

  if e.code == 0 { give Diagnostic.Info(0) }

  if e.severity <= 2 { give Diagnostic.Warn(e.code) }

  give Diagnostic.Error(e.code)
}

// Redaction: borne la charge utile avant diffusion
proc redact(e: Event) -> int {

  if e.payload_len < 0 { give 81 }

  if e.payload_len > 4096 { give 82 }

  give 0
}

proc handle(e: Event) -> int {
  let r: int = redact(e)

  if r != 0 { give r }
  let d: Diagnostic = classify(e)

  match d {
    case Info(_) { give 0 }
    case Warn(_) { give 0 }
    case Error(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let e: Event = Event(17, 3, 120)

  return handle(e)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.
  • classify: reads e: Event and returns Diagnostic.
  • redact: reads e: Event and returns int.
  • handle: reads e: Event and returns int.

2. Actual execution order

  1. The program enters main.
  2. handle is called to process the next step.
  3. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: e.code == 0.
  • Test evaluated: e.severity <= 2.
  • Test evaluated: e.payload_len < 0.
  • Test evaluated: e.payload_len > 4096.
  • Test evaluated: r != 0.
  • Selection by match d: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> handle -> sortie 0.
  • Example error: entrée invalide -> handle renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Info or warn level -> output 0.
  • Business error code 17 -> exit 17.
  • Payload out of range -> exit 82.

7. Line by line (variables + values)

Convenient reading: Follow the variables in the actual order of execution, then check the observed output.

  • Entry point:
  • entry main at core/app launches the complete scenario.
  • Block functions:
  • classify reads e: Event then returns Diagnostic.
  • redact reads e: Event then returns int.
  • handle reads e: Event then returns int.
  • Variables created (initial value):
  • r: int starts with redact(e).
  • d: Diagnostic starts with classify(e).
  • e: Event starts with Event(17, 3, 120).
  • Conditions that change the path:
  • if e.code == 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if e.severity <= 2 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if e.payload_len < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if e.payload_len > 4096 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if r != 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • Nominal trace (example values):
  • initialization: r=redact(e) -> d=classify(e) -> e=Event(17, 3, 120)
  • sequence: handle
  • final output on this path: handle(e).
  • Error trace (example values):
  • if e.code == 0 becomes true, the function returns Diagnostic.Info(0).
  • Quick check:
  • restart with normal input and note the output,
  • restart with an invalid entry and note the error code,
  • confirm that the same input always produces the same output.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
