Chapter pipeline diagram:

  • Invariant: same input + same state => same output.

26.0 Minimum editorial rule

Rule retained:

  • Document is an ordered list of lines.
  • Cursor always remains limited to the document.
  • Any mutant command is atomic (all or nothing).
  • undo/redo relies on fairly rich commands for exact reversal.
  • Any new order cancels future.

Invariants:

  • row in [0, max_row].
  • col in [0, len(line[row])].
  • past contains executed history; future contains only canceled orders.
  • After a new valid mutation: future = [].

26.1 Canonical model and EditResult

Border: this block does not know the terminal; it only models state and transitions.

form Cursor {
  row: int
  col: int
}

form Document {
  lines: string[]
}

form EditorState {
  doc: Document
  cursor: Cursor
  read_only: bool
}

pick EditorError {
  case ErrReadOnly
  case ErrInvalid
}

pick EditResult {
  case Ok(state: EditorState)
  case Err(error: EditorError)
}

pick Command {
  case Left
  case Right
  case InsertChar(ch: string, row: int, col: int)
  case Backspace(row: int, col: int, deleted: string)
  case NewLine(row: int, col: int)
}

form History {
  past: Command[]
  future: Command[]
}

Stability box:

This model is canonical and will no longer change in this chapter. All subsequent sections only add execution and verification rules.

Why this block exists:

  1. EditResult requires explicit processing of business errors instead of implicit errors.
  2. Backspace(.., deleted) captures the minimum information to guarantee an exact undo.
  3. History makes the editing trajectory verifiable and therefore testable.

Quick check: what happens if the input is invalid? Expected response: Err(ErrInvalid), without partial mutation.

26.2 Terminals and standardization

Border: this block imposes position constraints before any mutation.

form Bounds {
  max_row: int
  max_col: int
}

pub proc bounds_from_doc(d: Document) -> Bounds {
  let last_row: int = d.lines.len() - 1

  if last_row < 0 { give Bounds(0, 0) }
  let last_col: int = d.lines[last_row].len()

  give Bounds(last_row, last_col)
}

pub proc clamp_cursor(c: Cursor, d: Document) -> Cursor {
  let b: Bounds = bounds_from_doc(d)
  let r: int = c.row
  let k: int = c.col
  if r < 0 { set r = 0 }
  if r > b.max_row { set r = b.max_row }
  let line_len: int = d.lines[r].len()
  if k < 0 { set k = 0 }
  if k > line_len { set k = line_len }

  give Cursor(r, k)
}

26.3 Multi-line editing core

Border: this block applies text mutations; it does not manage history or rendering.

pub proc insert_at(s: EditorState, ch: string, row: int, col: int) -> EditorState {
  let c: Cursor = clamp_cursor(Cursor(row, col), s.doc)
  let line: string = s.doc.lines[c.row]
  let left: string = line.slice(0, c.col)
  let right: string = line.slice(c.col, line.len())
  s.doc.lines[c.row] = left + ch + right

  give EditorState(s.doc, Cursor(c.row, c.col + 1), s.read_only)
}

pub proc newline_split(s: EditorState, row: int, col: int) -> EditorState {
  let c: Cursor = clamp_cursor(Cursor(row, col), s.doc)
  let line: string = s.doc.lines[c.row]
  let left: string = line.slice(0, c.col)
  let right: string = line.slice(c.col, line.len())
  s.doc.lines[c.row] = left
  s.doc.lines.insert(c.row + 1, right)

  give EditorState(s.doc, Cursor(c.row + 1, 0), s.read_only)
}

pub proc backspace_apply(s: EditorState, row: int, col: int) -> EditorState {
  let c: Cursor = clamp_cursor(Cursor(row, col), s.doc)
  if c.col > 0 {
    let line: string = s.doc.lines[c.row]
    let left: string = line.slice(0, c.col - 1)
    let right: string = line.slice(c.col, line.len())
    s.doc.lines[c.row] = left + right

    give EditorState(s.doc, Cursor(c.row, c.col - 1), s.read_only)
  }

  if c.row == 0 { give s }

  let prev: string = s.doc.lines[c.row - 1]
  let cur: string = s.doc.lines[c.row]
  let join_col: int = prev.len()
  s.doc.lines[c.row - 1] = prev + cur
  s.doc.lines.remove_at(c.row)

  give EditorState(s.doc, Cursor(c.row - 1, join_col), s.read_only)
}

26.4 Exact backspace and formal inversion

Frontier: this block calculates exact inverses; it does not project the view.

Principle:

  • Backspace(row, col, deleted) stores what was deleted.
  • If intra-line deletion: deleted contains the removed character.
  • If merging lines: deleted contains " " to represent the deleted break.
pub proc inverse_of(cmd: Command) -> Command {

  match cmd {
    case Left { give Right }
    case Right { give Left }
    case InsertChar(ch, row, col) {
      give Backspace(row, col + ch.len(), ch)
    }
    case NewLine(row, col) {
      give Backspace(row + 1, 0, "
")
    }
    case Backspace(row, col, deleted) {
      if deleted == "
" {
        give NewLine(row - 1, col)
      }

      give InsertChar(deleted, row, col - deleted.len())
    }
    otherwise { give Left }
  }
}

Result: the inversion of Backspace no longer depends on a ? nor on a heuristic.

Single order table:

OrderPreconditionEffectReverse
LeftCursor bounded.Moves the cursor one column to the left (with clamp).Right.
RightCursor bounded.Moves the cursor one column to the right (with clamp).Left.
InsertChar(ch,row,col)ch not empty, normalized position.Insert ch to (row,col), advance cursor.Backspace(row, col + len(ch), ch).
Backspace(row,col,deleted)Normalized position, deleted conforms to deletion.Left delete or merge two lines if col == 0.InsertChar(deleted, row, col - len(deleted)) or NewLine(row - 1, col) if deleted == "\n".
NewLine(row,col)Standardized position.Cut the line in two and place the cursor at the start of the next line.Backspace(row + 1, 0, "\n").

26.5 Typed application and atomicity

Boundary: This block validates and applies a command with explicit return.

pub proc is_mutating(cmd: Command) -> bool {

  match cmd {
    case InsertChar(_, _, _) { give true }
    case Backspace(_, _, _) { give true }
    case NewLine(_, _) { give true }
    otherwise { give false }
  }
}

pub proc apply_command(s: EditorState, cmd: Command) -> EditResult {
  if s.doc.lines.len() == 0 {
    give Err(ErrInvalid)
  }

  if s.read_only and is_mutating(cmd) {
    give Err(ErrReadOnly)
  }

  match cmd {
    case Left {
      let c: Cursor = clamp_cursor(Cursor(s.cursor.row, s.cursor.col - 1), s.doc)

      give Ok(EditorState(s.doc, c, s.read_only))
    }
    case Right {
      let c: Cursor = clamp_cursor(Cursor(s.cursor.row, s.cursor.col + 1), s.doc)

      give Ok(EditorState(s.doc, c, s.read_only))
    }
    case InsertChar(ch, row, col) { give Ok(insert_at(s, ch, row, col)) }
    case Backspace(row, col, _) { give Ok(backspace_apply(s, row, col)) }
    case NewLine(row, col) { give Ok(newline_split(s, row, col)) }
    otherwise { give Err(ErrInvalid) }
  }
}

Atomicity rule:

  • Err(..) => initial state intact.
  • Ok(state) => command fully applied.

Concrete examples of refusal:

  • ErrInvalid: doc.lines = [] then apply_command(s, Right) returns Err(ErrInvalid).
  • ErrReadOnly: read_only = true then apply_command(s, InsertChar("x", 0, 0)) returns Err(ErrReadOnly).

26.6 Complete Undo/Redo and historical consistency

Border: this block handles historical + engine; it doesn't render.

pub proc push_history(h: History, cmd: Command) -> History {
  h.past.push(cmd)
  h.future = []

  give h
}

pub proc apply_with_history(s: EditorState, h: History, cmd: Command) -> EditResult {
  let r: EditResult = apply_command(s, cmd)

  match r {
    case Ok(next) {
      let h2: History = push_history(h, cmd)

      give Ok(next)
    }
    case Err(e) { give Err(e) }
    otherwise { give Err(ErrInvalid) }
  }
}

pub proc undo_step(s: EditorState, h: History) -> EditResult {

  if h.past.len() == 0 { give Ok(s) }
  let cmd: Command = h.past[h.past.len() - 1]
  h.past.remove_at(h.past.len() - 1)
  let inv: Command = inverse_of(cmd)
  let r: EditResult = apply_command(s, inv)

  match r {
    case Ok(next) {
      h.future.push(cmd)

      give Ok(next)
    }
    case Err(e) { give Err(e) }
    otherwise { give Err(ErrInvalid) }
  }
}

pub proc redo_step(s: EditorState, h: History) -> EditResult {

  if h.future.len() == 0 { give Ok(s) }
  let cmd: Command = h.future[h.future.len() - 1]
  h.future.remove_at(h.future.len() - 1)
  let r: EditResult = apply_command(s, cmd)

  match r {
    case Ok(next) {
      h.past.push(cmd)

      give Ok(next)
    }
    case Err(e) { give Err(e) }
    otherwise { give Err(ErrInvalid) }
  }
}

Explicit historical consistency invariant:

  • any new valid empty command future.
  • undo moves exactly one command from past to future.
  • redo does the opposite movement.

26.7 Complete multi-line scenario step by step

Initial document:

  • Line 0: "abc".
  • Cursor: (0, 3).

Step 1. NewLine(0, 3):

  • Before: "abc".
  • After: line 0 "abc", line 1 "".
  • Cursor: (1, 0).

Step 2. InsertChar("x", 1, 0):

  • Before: line 1 empty.
  • After: line 1 "x".
  • Cursor: (1, 1).

Step 3. Backspace(1, 0, " ") (merge):

  • Before: line 0 "abc", line 1 "x".
  • After: single line "abcx".
  • Cursor: (0, 3) (junction position).

Step 4. undo_step:

  • Reverse merge: NewLine(0, 3).
  • Return to two lines: "abc" and "x".

Step 5. redo_step:

  • Reapply the fusion.
  • Stable return to "abcx".

Multi-line trap test (intermediate empty line):

  • Initial document: line 0 "ab", line 1 "", line 2 "cd", cursor (2, 0).
  • Command: Backspace(2, 0, " ").
  • Expected: line 1 becomes "cd", line 2 is deleted, cursor on (1, 0).
  • no data lost despite the intermediate empty line.

26.8 Property test undo(redo(s)) == s

Subset of valid commands:

  • commands from a consistent state;
  • no row/col out of bounds after normalization;
  • compatible history (redo only if future not empty).

Pseudo-test:

pub proc prop_undo_redo_identity(s: EditorState, h: History) -> bool {
  let a: EditResult = redo_step(s, h)

  match a {
    case Ok(s1) {
      let b: EditResult = undo_step(s1, h)

      match b {
        case Ok(s2) { give hash_state(s2) == hash_state(s) }
        otherwise { give false }
      }
    }
    case Err(_) { give true }
    otherwise { give false }
  }
}

Goal: prove that a sequence redo then undo preserves state when it is applicable.

26.9 Long stable replay (50+ commands)

pub proc replay(s0: EditorState, cmds: Command[]) -> EditResult {
  let i: int = 0
  let s: EditorState = s0

  // Boucle: progression controlee jusqu'a la borne
  loop {
    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= cmds.len() { break }
    let r: EditResult = apply_command(s, cmds[i])

    match r {
      case Ok(next) { set s = next }
      case Err(e) { give Err(e) }
      otherwise { give Err(ErrInvalid) }
    }
    set i = i + 1
  }

  give Ok(s)
}

Recommended scenario:

  • 60 orders mixed (Left, Right, InsertChar, NewLine, Backspace).
  • Check final state, final cursor, past/future, then replay exactly the same sequence.
  • Expected: strictly identical final state (same hash).

26.10 Complexity and migration to rope/gap buffer

Current costs (row array + strings):

  • insert_at: O(L) on the active line.
  • backspace_apply: O(L) in local deletion; O(Lprev + Lcur) in fusion.
  • newline_split: O(L) on the broken line.

When to migrate:

  • very voluminous documents;
  • frequent insertions in the middle of long lines;
  • noticeable interactive latency.

Options:

  • gap buffer: simple for local editing around the cursor.
  • rope: best for large text and repeated concatenations/cuts.

API impact:

  • keep Document opaque and expose primitives (insert, delete, split, join).
  • preserve commands and EditResult; only the internal functions change.

Future Compatibility Box:

  • do not change the public form of Command.
  • do not change the semantics of EditResult.
  • keep the invariant new command => future = [].
  • preserve the behavior of inverse_of.
  • keep visible_lines pure (without mutation).

26.11 Concurrency and threads

Current status:

  • single-threaded, stable, serialized engine.

Recommended rule:

  • all orders go through a single queue.
  • a single worker applies Validate -> Apply -> Record.
  • projection reads immutable snapshots.

Benefit:

  • no race condition on doc/cursor/history.
  • replay and debug remain reproducible.

26.12 Minimal event log

Event:

form EditEvent {
  command: Command
  before_hash: string
  after_hash: string
  ts: int
}

Use:

  • audit: understand a discrepancy.
  • replay: replay a timeline.
  • debug: locate the transition that breaks an invariant.

Minimum recommended format:

  • one event per accepted order.
  • no event if Err(..).

Stable text log format (one line per event):

  • ts=<unix_ms> before=<hash> after=<hash> cmd=<CommandCompact>.
  • Example: ts=1710000123456 before=a91f after=b172 cmd=InsertChar(x,0,3).

26.13 Pure View Projection

form Viewport {
  top: int
  height: int
}

pub proc visible_lines(d: Document, v: Viewport) -> string[] {
  let out: string[] = []
  let i: int = v.top

  // Boucle: progression controlee jusqu'a la borne
  loop {
    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= d.lines.len() { break }

    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= v.top + v.height { break }
    out.push(d.lines[i])
    set i = i + 1
  }

  give out
}

Rule: projection never mutates Document.

26.14 Testable contracts

Exact assertions to automate:

  1. Terminals: after any order accepted, 0 <= row <= max_row and 0 <= col <= len(line[row]).
  2. Atomicity: if apply_command returns Err(_), the state hash remains the same.
  3. Undo/redo symmetry: on a compatible history, undo(redo(s)) == s.
  4. History: after any new valid mutant command, future.len() == 0.
  5. Projection purity: visible_lines never modifies the hash of Document.
  6. Replay determinism: replaying the same script produces exactly the same final hash.

26.15 Editor-specific final checklist

Ending checklist:

  • cursor terminals validated at each input.
  • guaranteed atomicity (Ok or Err, never partial state).
  • undo/redo symmetrical and tested.
  • read-only returns ErrReadOnly for mutation.
  • consistent history (new command => future = []).
  • strictly pure projection.
  • long stable replay validated.
  • event log enabled.

26.16 Design errors to avoid

Explicit anti-patterns:

  • duplicate validation in multiple layers instead of a single point of truth.
  • HTTP-like output mapping too early, before business state stabilization.
  • hidden mutation in the projection (visible_lines which would write in doc).
  • incomplete inversion of commands (absence of deleted for Backspace).
  • direct concurrent writing in doc/cursor/history without serialized file.

Error table -> diagnosis -> correction

ErrorDiagnosisCorrection
Invalid entryValidation missing or too late.Centralize pipeline input validation.
Inconsistent statePartial mutation or unverified invariant.Apply the principle of atomicity and reject otherwise.
Unexpected exitProjection layer mixed with business logic.Separate apply (job) and project (output).

Keep in mind

A robust editor isn't just about text rendering; it is a strict, typed, atomic and replayable state machine. The enriched Command + EditResult couple makes errors explicit, makes undo/redo more reliable and prepares the evolution towards more efficient structures without breaking the business API.

Quick check

Question: What happens if the entry is invalid? Expected response: Err(ErrInvalid) and status unchanged. If read_only is active on a mutation, Err(ErrReadOnly).

Final micro challenge: text selection

Add a select -> transform -> apply -> project pipeline without breaking the existing core.

Constraints:

  • select calculates a range (start, end) without mutating the document.
  • transform produces a compound command (ex: upper/lower/delete range).
  • apply remains atomic and returns EditResult.
  • project exposes the selection and result without hidden business logic.

5-step implementation plan

  1. MVP: canonical model + bounds + basic commands (Left, Right, InsertChar, NewLine, Backspace).
  2. Exact undo: enrich Backspace with deleted and finalize inverse_of.
  3. Replay: add full history + long stable tests.
  4. Observability: plug in EditEvent and stable logs one line per transition.
  5. Structural optimization: migrate internal to gap buffer or rope without breaking the public API.

// Scenario projet editor: execution complete et verifiable

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
pub proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
pub proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
pub proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
pub proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
