14. Macros (pocket version)

Level: Intermediate

Prerequisites: previous chapter docs/book/chapters/13-generics.html and docs/book/glossary.html. See also: docs/book/chapters/13-generics.html, docs/book/chapters/15-pipeline.html, docs/book/glossary.html.

Concrete Problem

Real situation: Macros become clear when we trace each step of the calculation. The goal is to connect line of code and concrete effect on the output. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: understand why a line exists and what it changes in the trajectory of the program.

Goal

You need to be able to reread a snippet, predict its outcome, and then verify that prediction with simple execution.

Why it matters

This chapter gives you a clear understanding of Macros. You will find the framework, the invariants and the reading decisions useful in practice.

What you're actually going to do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

Recommended first reflex: read the entries and conditions first, then only the syntactic form.

Reading method

  1. Identify the intent of the block.
  2. Check the main condition or test.
  3. Confirm the observable output.
  4. Note how this block serves Macros throughout the chapter.

Common pitfalls

  • Read the syntax without checking the behavior.
  • Mix general rule and error cases in the same explanation.
  • Introduce an optimization before having stabilized the flow of Macros.

Short exercise

Take an example from the chapter on Macros. Change a condition or input value, then check if the result remains as expected.

Summary in 5 points

  1. You know the objective of the chapter on Macros.
  2. You know how to read an example of the chapter in a structured way.
  3. You distinguish nominal case and error case.
  4. You avoid the most common pitfalls.
  5. You can reuse these rules in the next chapter.

14.1 Start with a deliberately neutral macro

macro nop() {
  let _: int = 0
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. No critical branching here: the flow is sequential, line by line.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: This block prepares a state, the output is calculated further in the pipeline.
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

The intention of this step is straightforward: start with a neutral macro to understand the expansion mechanism without business noise.

This progression avoids a common pitfall: writing powerful macros too early, before having stabilized the team's reading conventions.

At runtime, calling nop() does not add any functional logic. The observable flow of the program remains the same, which helps isolate the behavior of the macro itself.

Classic mistakes to avoid:

  • introduce a macro without being able to explain its expansion.
  • use a macro where a clear procedure is sufficient.
  • multiply the "magic" macros which hide the real flow.

14.2 Inject a reusable test into a sensitive procedure

macro guard_nonzero(x) {
  if x == 0 { return -1 }
}
proc safe_div(num: int, den: int) -> int {
  guard_nonzero(den)

  give num / den
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: x == 0. If true, the test path is executed immediately.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns -1.
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

The intent of this step is straightforward: use a macro to factor a repeated critical test.

The value of guard_nonzero is not just about saving a few lines. It imposes a uniform policy on a concrete risk: division by zero.

At runtime:

  • safe_div(10,2) passes the test and returns 5.
  • safe_div(10,0) is stopped by the test and returns -1.

The error boundary is placed exactly where the risk appears.

Classic mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

14.3 Encapsulating local standardization without dissolving responsibility

macro clamp01(v) {
  if v < 0 { set v = 0 }
  if v > 1 { set v = 1 }
}
proc normalize01(x: int) -> int {
  let v: int = x
  clamp01(v)

  give v
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: v < 0. If true, the test path is executed immediately.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns v.
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

The intent of this step is straightforward: to encapsulate local standardization without obscuring business responsibility.

The rule of thumb is simple: a macro is acceptable as long as a reader can reconstruct its expansion without disproportionate effort.

At runtime:

  • normalize01(-3) returns 0.
  • normalize01(0) returns 0.
  • normalize01(5) returns 1.

The output is bounded in [0,1], and this guarantee remains visible in the source logic.

Classic mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

Keep in mind

A useful Vitte macro is neither mystical nor decorative. It makes a recurring rule more stable, more auditable and easier to evolve. If it obscures the intent, remove it. If it clarifies a repeated rule, keep it and document it like a piece of architecture. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

Practical quality criterion for this chapter:

  • you can explain macro expansion in a few sentences.
  • you know why the macro exists instead of a procedure.
  • you can modify the macro rule without breaking existing calls.

Quick check

Question: What happens if the entry is invalid? Benchmark: An explicit test or stable fallback path must apply.

Try this

  1. Take an example from the chapter and modify a test condition to observe different behavior.
  2. Write a mini mental test on an invalid entry in the chapter, then predict the executed branch.

If you're stuck

  • identify the modified line and explain in one sentence the new expected output.
  • Name the test or fallback branch actually used.

Extended Example

// Scenario macros: execution complete et verifiable
space demo/macros

form Input { id: int value: int quota: int }
pick Eval { case Accepted(score: int) case Rejected(code: int) }

proc validate(x: Input) -> Eval {

  if x.id <= 0 { give Eval.Rejected(21) }

  if x.quota < 0 { give Eval.Rejected(22) }

  if x.value < 0 { give Eval.Rejected(23) }

  give Eval.Accepted(x.value)
}

proc transform(score: int, quota: int) -> int {
  let capped: int = score
  if capped > quota { set capped = quota }

  if capped < 0 { give 0 }

  give capped * 2
}

proc decide(r: Eval, quota: int) -> Eval {

  match r {
    case Accepted(s) {
      let out: int = transform(s, quota)

      if out >= 10 { give Eval.Accepted(out) }

      give Eval.Rejected(31)
    }
    case Rejected(c) { give Eval.Rejected(c) }
    otherwise { give Eval.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(r: Eval) -> int {

  match r {
    case Accepted(_) { give 0 }
    case Rejected(code) { give code }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let x: Input = Input(1, 8, 9)
  let v: Eval = validate(x)
  let d: Eval = decide(v, x.quota)

  return to_exit(d)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • validate: reads x: Input and returns Eval.
  • transform: reads score: int, quota: int and returns int.
  • decide: reads r: Eval, quota: int and returns Eval.
  • to_exit: reads r: Eval and returns int.

2. Actual execution order

  1. The program enters main.
  2. validate is called to process the next step.
  3. decide is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: x.id <= 0.
  • Test evaluated: x.quota < 0.
  • Test evaluated: x.value < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 0.
  • Test evaluated: out >= 10.
  • Selection by match r: the path depends on the state received.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> validate -> decide -> to_exit -> sortie 0.
  • Example error: entrée invalide -> validate renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal case -> output 0.
  • Strict quota case -> stable behavior.
  • Invalid case id<=0 -> output 21.

7. Line by line (variables + values)

Convenient reading: Follow the variables in the actual order of execution, then check the observed output.

  • Entry point:
  • entry main at core/app launches the complete scenario.
  • Block functions:
  • validate reads x: Input then returns Eval.
  • transform reads score: int, quota: int then returns int.
  • decide reads r: Eval, quota: int then returns Eval.
  • to_exit reads r: Eval then returns int.
  • Variables created (initial value):
  • capped: int starts with score.
  • out: int starts with transform(s, quota).
  • x: Input starts with Input(1, 8, 9).
  • v: Eval starts with validate(x).
  • d: Eval starts with decide(v, x.quota).
  • Variables changed during processing:
  • capped is updated with quota.
  • Conditions that change the path:
  • if x.id <= 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if x.quota < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if x.value < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if capped > quota is true: early exit or dedicated branch; otherwise: the flow continues.
  • if capped < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if out >= 10 is true: early exit or dedicated branch; otherwise: the flow continues.
  • Nominal trace (example values):
  • initialization: capped=score -> out=transform(s, quota) -> x=Input(1, 8, 9) -> v=validate(x)
  • sequence: validate -> decide -> to_exit
  • final output on this path: to_exit(d).

  • if x.id <= 0 becomes true, the function returns Eval.Rejected(21).

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
