This chapter pursues a clear objective: to build a Vitte on-board loop usable in real conditions, with sensor noise management, robust thresholds, actuator safety and stable behavior. We're going to move from a simple demonstration to a complete mini controller.

The guiding idea is the following: in embedded, robustness comes from the discipline of invariants. Each step must be explicit, testable and isolable: acquisition, sanitation, filtering, decision, material projection.

Reference: see the Vitte Glossary in docs/book/glossary.html and the Editorial Checklist in docs/book/checklist.html. Complement: docs/book/classic-mistakes.html.

Chapter pipeline diagram:

  • Input: initial data or query.
  • Processing: validation, business logic, transformations.
  • Output: business result or status code.
  • Invariant: property that must remain true at each step.

25.2 Acquire and saturate the raw measurement

pub proc read_raw(v: int) -> Sample {
  give Sample(v)
}
pub proc clamp_raw(s: Sample, cfg: ControllerCfg) -> int {

  if s.raw < cfg.min_raw { give cfg.min_raw }

  if s.raw > cfg.max_raw { give cfg.max_raw }

  give s.raw
}

Saturation must be the first robustness barrier:

  • it cuts impossible values.
  • it stabilizes the following stages.

At runtime:

  • value too low -> raised to min_raw.
  • value too high -> reduced to max_raw.

Common mistakes to avoid:

  • normalize an unsaturated value.
  • assume a fixed range (0.1023) while the map changes.
  • treat excesses in a late layer.

25.4 Normalize to calibrated percentage

pub proc to_percent(v: int, cfg: ControllerCfg) -> int {
  let span: int = cfg.max_raw - cfg.min_raw

  if span <= 0 { give 0 }

  give ((v - cfg.min_raw) * 100) / span
}

25.5 Classify with hysteresis (anti-flashing)

pub proc classify_hysteresis(p: int, prev: ControlState, cfg: ControllerCfg) -> ControlState {

  match prev {
    case Alert {
      if p <= cfg.alert_off { give Armed }

      give Alert
    }
    case Armed {
      if p >= cfg.alert_on { give Alert }

      give Armed
    }
    case Idle {
      if p >= cfg.alert_on { give Alert }

      give Armed
    }
    case Fault(code) { give Fault(code) }
    otherwise { give Fault(900) }
  }
}

Common mistakes to avoid:

  • use a single threshold and suffer the flashing.
  • forget the previous state in the decision.
  • force the exit of Fault without diagnostics.

25.6 Detect sensor faults

pub proc detect_fault(raw: int, cfg: ControllerCfg) -> ControlState {

25.7 Compose a complete controller iteration

form ControllerState {
  filter: Filter3
  control: ControlState
  percent: int
}
pub proc step(raw_input: int, st: ControllerState, cfg: ControllerCfg) -> ControllerState {
  let s: Sample = read_raw(raw_input)
  let f0: ControlState = detect_fault(s.raw, cfg)

  match f0 {
    case Fault(code) {
      give ControllerState(st.filter, Fault(code), st.percent)
    }
    otherwise {
      let clean: int = clamp_raw(s, cfg)
      let f1: Filter3 = filter3_push(st.filter, clean)
      let avg: int = filter3_mean(f1)
      let p: int = to_percent(avg, cfg)
      let next_ctrl: ControlState = classify_hysteresis(p, st.control, cfg)

      give ControllerState(f1, next_ctrl, p)
    }
  }
}

25.8 Project to actuator in safe mode

25.10 Ultra concrete test plan

  1. Saturation tests:
  2. raw < min_raw => clamp to min_raw.
  3. raw > max_raw => clamp to max_raw.
  1. Filter tests:
  2. push 100,100,100 => mean 100.
  3. push 0,0,1023 => expected stable average.
  1. Hysteresis tests:
  2. Armed + p=81 => Alert.
  3. Alert + p=75 => remains Alert.
  4. Alert + p=69 => Armed.
  1. Fault Tests:
  2. raw very low => Fault(1001).
  3. raw very high => Fault(1002).
  4. in Fault, PWM must remain 0.
  1. Replay tests:
  2. same sequence of inputs => same states and same outputs.

25.11 Production roadmap

To go to real firmware:

  1. Persistent EEPROM/flash calibration.
  2. Watchdog + resume code on reboot.
  3. Serial telemetry (state, raw, filtered, pwm).
  4. Offline simulation mode to replay captures.
  5. Energy profile (loop frequency, sleep).

Error table -> diagnosis -> correction

ErrorDiagnosisCorrection
Invalid entryValidation missing or too late.Centralize pipeline input validation.
Inconsistent statePartial mutation or unverified invariant.Apply the principle of atomicity and reject otherwise.
Unexpected exitProjection layer mixed with business logic.Separate apply (job) and project (output).

Keep in mind

A robust embedded controller is about more than a threshold. It combines calibration, filtering, hysteresis, explicit fault management and fail-safe projection. Code becomes maintainable when each stage of the pipeline is isolated, testable and replayable.

Practical quality criterion for this chapter:

  • you know how to explain the complete pipeline from raw_input to pwm.
  • you can prove that the output is safe in case of Fault.
  • you can replay a sensor scenario and find exactly the same state transitions.

Extended Example

// Scenario projet arduino: execution complete et verifiable
space demo/projet-arduino

form Telemetry { temp_c: int volts_mv: int seq: int }
pick Decision { case Keep case Cool(level: int) case Fault(code: int) }

pub proc read_sensor(step: int) -> Telemetry {
  let t: int = 30 + (step % 10)
  let v: int = 3300 - (step % 30)

  give Telemetry(t, v, step)
}

pub proc validate(t: Telemetry) -> int {

  if t.volts_mv < 3000 { give 61 }

  if t.temp_c < -20 { give 62 }

  if t.temp_c > 120 { give 63 }

  give 0
}

pub proc control(t: Telemetry) -> Decision {
  let v: int = validate(t)

  if v != 0 { give Decision.Fault(v) }

  if t.temp_c >= 36 { give Decision.Cool(2) }

  if t.temp_c >= 33 { give Decision.Cool(1) }

  give Decision.Keep
}

// Conversion finale vers un code de sortie
pub proc to_exit(d: Decision) -> int {

  match d {
    case Keep { give 0 }
    case Cool(_) { give 0 }
    case Fault(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let t: Telemetry = read_sensor(9)
  let d: Decision = control(t)

  return to_exit(d)
}

Detailed explanation of the big block

You read this big block as a complete scenario: data preparation, processing, then final output.

1. Role of each party

  • Starting point: entry main at core/app.
  • read_sensor: reads step: int and returns Telemetry.
  • validate: reads t: Telemetry and returns int.
  • control: reads t: Telemetry and returns Decision.
  • to_exit: reads d: Decision and returns int.

2. Actual execution order

  1. The program enters main.
  2. read_sensor is called to process the next step.
  3. control is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: t.volts_mv < 3000.
  • Test evaluated: t.temp_c < -20.
  • Test evaluated: t.temp_c > 120.
  • Test evaluated: v != 0.
  • Test evaluated: t.temp_c >= 36.
  • Test evaluated: t.temp_c >= 33.
  • Selection by match d: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> read_sensor -> control -> to_exit -> sortie 0.
  • Example error: entrée invalide -> read_sensor renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • isolate a real error case and predict its exit,

What I would do in code review
