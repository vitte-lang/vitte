This chapter pursues a clear objective: to build a realistic HTTP service by strictly separating transport, business routing and final status projection. The goal is not to stack conditions, but to lay down a stable, testable and traceable pipeline.

Reference: see the Vitte Glossary in docs/book/glossary.html and the Editorial Checklist in docs/book/checklist.html. Complement: docs/book/classic-mistakes.html.

Chapter pipeline diagram:

  • Request -> Validate -> Route -> Execute -> Map Response.
  • Input: raw request (method, path, payload, token).
  • Processing: transport validation then business routing.
  • Output: HttpResult then final HTTP code.
  • Invariant: a given error always produces the same status.

22.0 Minimum HTTP rule

Rule retained for this chapter:

  • Method: GET only (other methods are rejected).
  • Supported paths: /health, /metrics.
  • Payload: length body_len >= 0.
  • Authentication: auth_token not empty.
  • Expected status:
  • nominal /health or /metrics validated: 200.
  • invalid method or invalid payload: 400.
  • missing token: 401.
  • unknown path: 404.
  • unforeseen case: 500.

Standardized status table:

  • 400: cause requête invalide ; responsible layer validation transport ; corrective action corriger méthode/payload côté client.
  • 401: cause auth manquante/invalide ; responsible layer validation transport ; corrective action fournir un token valide.
  • 404: cause route absente ; responsible layer routage ; corrective action utiliser un chemin exposé.
  • 500: cause cas inattendu ; responsible layer projection finale ; corrective action journaliser + corriger la branche manquante.

22.1 Pose the transport and error model

Boundary: this block does not know HTTP at the socket/server level, it only models data and results.

form HttpRequest {
  method: string
  path: string
  body_len: int
  auth_token: string
}

pick HttpError {
  case BadRequest
  case Unauthorized
  case NotFound
}

pick HttpResult {
  case Ok(code: int)
  case Err(e: HttpError)
}

Reading line by line (beginner):

  1. form HttpRequest { opens the transport request structure used by the entire pipeline.
  2. method: string adds HTTP method (GET, POST, etc.) for input validation.
  3. path: string adds the route path to resolve.
  4. body_len: int adds payload length to detect invalid format.
  5. auth_token: string adds minimum authentication information.
  6. pick HttpError { opens the closed set of HTTP layer business errors.
  7. case BadRequest represents an invalid client-side request.
  8. case Unauthorized represents an authentication failure.
  9. case NotFound represents a missing route.
  10. pick HttpResult { encodes a uniform result for all steps.
  11. case Ok(code: int) carries an explicit nominal status.
  12. case Err(e: HttpError) carries a typed error for the final projection.

Input -> output (to be checked):

  • Error case: if a step detects a fault, it returns Err(..) rather than a magic integer.
  • Nominal case: a success transports Ok(code).
  • Testable observation: possible transitions are bounded by HttpResult.

Quick check: what happens if the input is invalid? Expected response: the value changes to Err(..) from the first test concerned.

22.2 Validate the request at the system edge

Boundary: This block projects input HTTP constraints, but does not yet decide the business logic.

pub proc validate_transport(r: HttpRequest) -> HttpResult {

  if r.method != "GET" { give Err(BadRequest) }

  if r.path == "" { give Err(BadRequest) }

  if r.body_len < 0 { give Err(BadRequest) }

  if r.auth_token == "" { give Err(Unauthorized) }

  give Ok(200)
}

Reinforced validation, concrete invalid entries:

  • method="POST" -> Err(BadRequest).
  • path="" -> Err(BadRequest).
  • auth_token="" -> Err(Unauthorized).

Reading line by line (beginner):

  1. if r.method != "GET" { give Err(BadRequest) } immediately rejects out-of-rule methods.
  2. if r.path == "" { give Err(BadRequest) } rejects an empty route.
  3. if r.body_len < 0 { give Err(BadRequest) } rejects inconsistent length.
  4. if r.auth_token == "" { give Err(Unauthorized) } rejects the absence of token.
  5. give Ok(200) authorizes the continuation of the pipeline when the transport is valid.

Input -> output (to be checked):

  • Error case: path="" gives Err(BadRequest).
  • Nominal case: GET /health with non-empty token gives Ok(200).
  • Testable observation: the same invalid query always produces the same error.

Quick check: what happens if the input is invalid? Expected response: the function stops on the first invalid test.

22.3 Isolate path routing

Border: this block does not know raw transport, it only routes an already validated path.

pub proc route_path(path: string) -> HttpResult {

  if path == "/health" { give Ok(200) }

  if path == "/metrics" { give Ok(200) }

  give Err(NotFound)
}

Reading line by line (beginner):

  1. if path == "/health" { give Ok(200) } nominal health route.
  2. if path == "/metrics" { give Ok(200) } nominal route of metrics.
  3. give Err(NotFound) unknown route explicitly typed.

Input -> output (to be checked):

  • Error case: /unknown gives Err(NotFound).
  • Nominal case: /health gives Ok(200).
  • Testable observation: routing is stable and independent of auth.

Quick check: what happens if the input is invalid? Expected response: Unknown route is converted to Err(NotFound).

22.4 Compose steps in a handler

Border: this block orchestrates transport + routing, without duplicating the local rules of each layer.

pub proc handle(r: HttpRequest) -> HttpResult {
  let t: HttpResult = validate_transport(r)

  match t {
    case Err(e) { give Err(e) }
    case Ok(_) { give route_path(r.path) }
    otherwise { give Err(BadRequest) }
  }
}

Reading line by line (beginner):

  1. let t: HttpResult = validate_transport(r) executes the transport boundary first.
  2. case Err(e) { give Err(e) } immediately shorts on failure.
  3. case Ok(_) { give route_path(r.path) } switches to routing only if transport is valid.
  4. otherwise { give Err(BadRequest) } locks a stable fallback.

Input -> output (to be checked):

  • Error case: token missing -> Err(Unauthorized) without going to routing.
  • Nominal case: valid transport + /metrics -> Ok(200).
  • Testable observation: no validation duplication in the router.

Quick check: what happens if the input is invalid? Expected response: Err(e) branch returns the already qualified error.

22.5 Convert result to final HTTP status

Border: this block projects in HTTP.

pub proc to_http_code(r: HttpResult) -> int {

  match r {
    case Ok(c) { give c }
    case Err(BadRequest) { give 400 }
    case Err(Unauthorized) { give 401 }
    case Err(NotFound) { give 404 }
    otherwise { give 500 }
  }
}

Reading line by line (beginner):

  1. case Ok(c) { give c } maintains nominal status.
  2. case Err(BadRequest) { give 400 } maps the client error to 400.
  3. case Err(Unauthorized) { give 401 } maps the invalid auth to 401.
  4. case Err(NotFound) { give 404 } maps absent route to 404.
  5. otherwise { give 500 } protects production with a safety net.

Input -> output (to be checked):

  • Error case: Unexpected error -> 500.
  • Nominal case: Ok(200) -> 200.
  • Testable observation: each known HttpError has a stable status.

Quick check: what happens if the input is invalid? Expected answer: it is translated into an explicit, unambiguous 4xx/5xx code.

22.6 Idempotence (operation rule)

Rule of this chapter:

  • GET /health: idempotent, no side effects, repeatable without change of state.
  • GET /metrics: idempotent in this model, read-only.
  • Transport validation: pure, without global mutation.

Practical consequence: replaying the same valid request produces the same status and does not modify the state of the service.

22.7 Complete traceable scenario

Mandatory journey:

  • Raw input: method="GET", path="/health", body_len=0, auth_token="t".
  • Validate: Ok(200).
  • Route: Ok(200).
  • Execute (handler): Ok(200).
  • Map Response: 200.

Invalid route (auth missing):

  • Raw input: auth_token="".
  • Validate: Err(Unauthorized).
  • Handler: short circuit Err(Unauthorized).
  • Map Response: 401.

22.8 Minimal integration tests

  1. Nominal:
  2. Request: GET /health, body_len=0, non-empty token.
  3. Expected: to_http_code(handle(r)) == 200.
  1. Auth fail:
  2. Request: GET /health, empty token.
  3. Expected: to_http_code(handle(r)) == 401.
  1. Route not found:
  2. Request: GET /unknown, non-empty token.
  3. Expected: to_http_code(handle(r)) == 404.

22.9 Classic HTTP errors

  • Mapping to status too early: we lose business information (Err(..)) and complicate debugging.
  • Duplicate validation in multiple layers: discrepancies and inconsistencies guaranteed.
  • Mixing transport and business in a single function: fragile tests, expensive maintenance.
  • Forget the fallback 500: unstable behavior in unexpected cases.

Error table -> diagnosis -> correction

ErrorDiagnosisCorrection
Invalid entryValidation missing or too late.Centralize pipeline input validation.
Inconsistent statePartial mutation or unverified invariant.Apply the principle of atomicity and reject otherwise.
Unexpected exitProjection layer mixed with business logic.Separate apply (job) and project (output).

Keep in mind

A robust HTTP service starts with clear boundaries: transport validation, independent routing, minimal orchestration, centralized final projection. If each layer tests its responsibility, the system becomes readable, testable and stable.

Practical quality criterion for this chapter:

  • you know how to immediately locate the layer responsible for an error.
  • you can replay a complete query flow without ambiguity.
  • you can add a route without breaking the status mapping.

Quick check

Question: What happens if the entry is invalid? Expected response: A validation test or fallback route converts the input to Err(..) and then to stable HTTP code.

Try this

  1. Add the GET /ready route while maintaining the validation/routing/projection separation.
  2. Add a rule body_len > 1_000_000 -> Err(BadRequest) and check the impact on tests.

If you're stuck

  • The new route must be added in route_path without modifying validate_transport.
  • The new size test must live in validate_transport and keep the projection centralized in to_http_code.

HTTP micro challenge

Add 429 Too Many Requests without breaking existing mappings.

Design constraint:

  • extend HttpError with a new case.
  • do not duplicate status conversion outside to_http_code.
  • ensure that 200/401/404 tests remain unchanged.

Extended Example

// Scenario projet http: execution complete et verifiable
space demo/projet-http

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
pub proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
pub proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
pub proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
pub proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

  • to_exit: reads x: Result and returns int.

  3. apply_policy is called to process the next step.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

To test:
