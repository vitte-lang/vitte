Keyword trait

Level: Advanced.

Quick reading

Concrete reference: trait names a specific language contract. Use it when you need this exact form; avoid it when a simpler construct carries the same rule.

Why (job)

In practice, trait should make the rule clearer, not noisier. If removing it changes neither the meaning nor the outcome, prefer the simpler form.

Definition

trait declares a behavior contract and can group proc signatures, constants and type aliases.

Syntax

Canonical forms:

  • trait Name { proc ready(self) -> bool }
  • unsafe trait Name { ... }
  • trait Name[T] where T: SomeTrait { ... }

The items allowed in the body are:

  • proc signature or with body
  • const
  • type alias

An implementation follows the form:

  • impl Trait for Type { ... }
  • impl Type { ... }

Algorithmic reading

Read it in context: find where trait appears, then confirm the constraint it adds.

Nominal example

Input:

  • Valid brace declaration.
// Exemple concret: cas nominal puis cas invalide
trait Pair {
  proc left(self) -> int
  proc right(self) -> int
}

impl Pair for BoxedPair {
  proc left(self) -> int { give 0 }
  proc right(self) -> int { give 1 }
}

Observed result:

  • The declaration conforms to the trait_decl syntax.
  • Items are recognized as trait signatures/methods, not fields.

Invalid example

Input:

  • Form outside surface grammar.
// Exemple concret: cas nominal puis cas invalide
trait Pair {
  proc left(self) -> int
  proc right(self)
}
# invalide: type manquant pour `right`.

Observed result:

  • The parser rejects the declaration because a trait proc signature must have a complete form.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable, actionable correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Projecting OO semantics not carried by grammar.
  • Put fields in a modern trait.
  • Forget the consistency between trait and impl.

When to use it / When to avoid it

  • When to use it: to describe a shared contract then specialize it with impl.
  • When to avoid it: If you only need a simple form.

Common compiler errors

Typical messageCauseCorrection
unexpected token near traitNon-compliant body (proc, const, type).Keep trait items valid and close the block correctly.
mixed declaration styleMixture of trait items and form fields.Use trait for contracts and form for data.
missing proc bodyAn ordinary method does not have a signature-only out-of-context body.Add a block or place it in a trait/impl.

Neighbor keyword

KeywordOperational difference
formform describes the data, trait describes the contract, impl provides the execution.
