Keyword pull

Level: Advanced.

Quick reading

Concrete reference: pull is used to decide an execution path in a readable and verifiable manner. Use pull when used to organize modules and symbols; avoid it if it makes no observable difference on the branch or output.

Why (job)

In algorithmic practice, pull is used to transform a condition into an executable branch, without reading ambiguity. Rule of thumb: if removing pull changes neither the executed path nor the output, simplify the block.

Definition

pull imports a top-level module, with optional alias.

Syntax

Canonical form: pull module/path [as alias].

Algorithmic reading

Recommended Reading: Track down where pull occurs in the flow, then check its effect on actual execution.

Nominal example

Entrance:

  • Top-level import with alias.
// Exemple concret: cas nominal puis cas invalide

space api/v1
pull core/math as math

proc sum(a: int, b: int) -> int {
  // Valeur retournee par cette branche
  give math.add(a, b)
}

Observable output:

  • Module core/math is resolved under the alias math.

Invalid example

Entrance:

  • Invalid syntactic order.
// Exemple concret: cas nominal puis cas invalide
pull as core/math
# invalide: `as` vient après le chemin module.

Observable output:

  • The declaration is rejected by the parser.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable pattern oriented correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Import with aliases too generic.
  • Declare pull in a block.
  • Use an undocumented relative path.

When to use it / When to avoid it

  • When to use it: to import a clearly named project module.
  • When to avoid it: in a block of statements or with an ambiguous alias.
