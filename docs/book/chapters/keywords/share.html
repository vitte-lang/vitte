Keyword share

Level: Advanced.

Quick reading

Concrete reference: share is used to decide an execution path in a readable and verifiable manner. Use share when used to organize modules and symbols; avoid it if it makes no observable difference on the branch or output.

Why (job)

In practice, share should make the rule clearer, not noisier. If removing it changes neither the meaning nor the outcome, prefer the simpler form.

Definition

share explicitly exports top-level symbols (all or list of identifiers).

Syntax

Canonical form: share all or share id1, id2.

Algorithmic reading

Read it in context: find where share appears, then confirm the constraint it adds.

Nominal example

Input:

  • Explicit export of symbols.
// Exemple concret: cas nominal puis cas invalide

space core/math

proc add(a: int, b: int) -> int { give a + b }

proc sub(a: int, b: int) -> int { give a - b }
share add, sub

Observed result:

  • Only add and sub are exported.

Invalid example

Input:

  • Non-grammar export form.
// Exemple concret: cas nominal puis cas invalide
share proc add(a: int, b: int) -> int { give a + b }
# invalide: `share` n’exporte pas une déclaration inline.

Observed result:

  • The parser rejects the shape.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable, actionable correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Export too wide with all without need.
  • Export unstable internal symbol.
  • Forgetting to align share with the module documentation.

When to use it / When to avoid it

  • When to use it: To control a module's public API.
  • When to avoid it: to implicitly export unstabilized elements.
