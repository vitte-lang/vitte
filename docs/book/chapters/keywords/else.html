Keyword else

Level: Beginner.

Quick reading

Concrete reference: else is used to decide an execution path in a readable and verifiable manner. Use else when it is used to drive a control decision; avoid it if it makes no observable difference on the branch or output.

Why (job)

In practice, else should make the rule clearer, not noisier. If removing it changes neither the meaning nor the outcome, prefer the simpler form.

Definition

else is a keyword in the Vitte language. This sheet provides operational use with a readable and testable rule.

Syntax

Canonical form: if cond { ... } else { ... }.

Algorithmic reading

Read it in context: find where else appears, then confirm the constraint it adds.

Nominal example

Input:

  • Nominal case: else is used in the correct context and the expected branch is executed.
// Exemple concret: cas nominal puis cas invalide

proc sign(x: int) -> int {
  // Test: condition explicite avant action
  if x < 0 { give -1 }
  else { give 1 }
}

Observed result:

  • For nominal input, the expected branch is taken and the output matches the expected calculation.

Invalid example

Input:

  • Error case: invalid use of else; validation must reject the snippet with an actionable diagnosis.
// Exemple concret: cas nominal puis cas invalide

proc bad(x: int) -> int {
  else { give x }
}
# invalide: `else` sans `if` precedent.

Observed result:

  • For invalid input, validation should fail with an actionable message to correct the code.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable, actionable correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Using else out of habit instead of justifying its role in the flow.
  • Mix business logic and control logic without an explicit boundary.
  • Forgetting to test a dedicated invalid case.

When to use it / When to avoid it

  • When to use it: when else makes the contract clearer.
  • When to avoid it: when its use hides the rule or duplicates logic already carried elsewhere.
