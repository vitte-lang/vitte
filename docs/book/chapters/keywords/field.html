Keyword field

Level: Intermediate.

Quick reading

Concrete reference: field is used to decide an execution path in a readable and verifiable manner. Use field when used to define a data model; avoid it if it makes no observable difference on the branch or output.

Why (job)

In practice, field should make the rule clearer, not noisier. If removing it changes neither the meaning nor the outcome, prefer the simpler form.

Definition

field is used in the legacy form of form with the syntax as.

Syntax

Canonical form: field nom as type (in legacy_form).

Algorithmic reading

Read it in context: find where field appears, then confirm the constraint it adds.

Nominal example

Input:

  • Valid legacy declaration.
// Exemple concret: cas nominal puis cas invalide

form User
field id as int
field name as string
.end

Observed result:

  • The structure is accepted with two typed fields.

Invalid example

Input:

  • Form mixing syntaxes.
// Exemple concret: cas nominal puis cas invalide

form User
field id: int
.end
# invalide: en mode legacy, `field` utilise `as`.

Observed result:

  • The parser rejects the field line.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable, actionable correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Use field with :.
  • Mix .end and { ... }.
  • Forget the migration to the modern form.

When to use it / When to avoid it

  • When to use it: to maintain legacy code .end.
  • When to avoid it: in the modern brace form (name: type).
