Concrete Problem

Real situation: in this chapter on Embedded Project (memory/time constraints), failure often comes from a poorly ordered decision rather than a syntax error. So we follow the exact flow: input, test, branch, output. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: each section uses the same scenario to isolate a single technical decision at a time. Educational objective: move from passive reading to proof: same input, same branch, same expected output.

Goal

Deliver an embedded project respecting memory budget and latency constraints.

Axes

  1. Strict memory profile.
  2. Deterministic critical time loops.
  3. Defensive management of I/O errors.
  4. Verification on real target.

Documentation to produce

  1. Memory budget (static/dynamic) per module.
  2. Time budget per critical loop.
  3. Failure mode matrix and expected behavior.
  4. Verification procedure on target (not only emulator).
  5. Minimum hardware constraints and supported versions.

Target budgets (example)

Sensor acquisition8 KB24 KB<= 2ms

Build rejection criteria

  1. Build KO if estimated RAM > overall budget.
  2. Build KO if estimated Flash > overall budget.
  3. Build KO if p95 latency > target on test bench.
  4. Build KO if an actual target check is missing.

Progressive examples (N1 -> N3)

N1 (base): single sensor loop

proc sample_once(v: int) -> int {
  give v
}

Goal:

  1. Minimum nominal path, without dynamic allocation.

N2 (intermediate): time/response test

proc read_checked(ok: bool, value: int) -> int {

  if not ok { give -1 }

  give value
}

Goal:

  1. Make the sensor failure path explicit.

N3 (advance): budget + system control

- RAM cible: <= 32 KB
- Flash cible: <= 96 KB
- boucle complete p95: <= 10 ms
- vérification sur carte reelle obligatoire

Goal:

  1. Check memory/latency constraints before release.

Anti-example (invalid embedded production)

- allocations non bornees en boucle
- aucune mesure p95
- validation uniquement sur emulateur

Why this is a problem:

  1. Risk of memory overflow.
  2. Non-deterministic latency risk.

Quick validation

  1. Check N1 (no unnecessary allocation).
  2. Check N2 (failure covered).
  3. Check N3 (budgets and measures respected).

Why it matters

This section explains the practical value: reducing errors, accelerating diagnosis and stabilizing developments.

Quick check

Control question: If you change a key assumption, what outcome should change and why?

Try this

  1. Run the nominal example.
  2. Introduce an error case.
  3. Check the output and document the deviation.

If you're stuck

Fixed: keep the simplest version that respects the rule, then add a non-regression test.

Extended Example

// Scenario projet embarque contraintes: execution complete et verifiable
space demo/projet-embarque-contraintes

form Telemetry { temp_c: int volts_mv: int seq: int }
pick Decision { case Keep case Cool(level: int) case Fault(code: int) }

proc read_sensor(step: int) -> Telemetry {
  let t: int = 30 + (step % 10)
  let v: int = 3300 - (step % 30)

  give Telemetry(t, v, step)
}

proc validate(t: Telemetry) -> int {

  if t.volts_mv < 3000 { give 61 }

  if t.temp_c < -20 { give 62 }

  if t.temp_c > 120 { give 63 }

  give 0
}

proc control(t: Telemetry) -> Decision {
  let v: int = validate(t)

  if v != 0 { give Decision.Fault(v) }

  if t.temp_c >= 36 { give Decision.Cool(2) }

  if t.temp_c >= 33 { give Decision.Cool(1) }

  give Decision.Keep
}

// Conversion finale vers un code de sortie
proc to_exit(d: Decision) -> int {

  match d {
    case Keep { give 0 }
    case Cool(_) { give 0 }
    case Fault(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let t: Telemetry = read_sensor(9)
  let d: Decision = control(t)

  return to_exit(d)
}

Detailed explanation of the big block

You read this big block as a complete scenario: data preparation, processing, then final output.

1. Role of each party

  • Starting point: entry main at core/app.
  • read_sensor: reads step: int and returns Telemetry.
  • validate: reads t: Telemetry and returns int.
  • control: reads t: Telemetry and returns Decision.
  • to_exit: reads d: Decision and returns int.

2. Actual execution order

  1. The program enters main.
  2. read_sensor is called to process the next step.
  3. control is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: t.volts_mv < 3000.
  • Test evaluated: t.temp_c < -20.
  • Test evaluated: t.temp_c > 120.
  • Test evaluated: v != 0.
  • Test evaluated: t.temp_c >= 36.
  • Test evaluated: t.temp_c >= 33.
  • Selection by match d: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> read_sensor -> control -> to_exit -> sortie 0.
  • Example error: entrée invalide -> read_sensor renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal temp -> output 0.
  • Undervoltage -> output 61.
  • Temperature out of range -> output 62 or 63.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Mini Case Study (Before / After)

Before: mixed business logic and technical output, expensive diagnosis. After: entrance tests, business decision, final screening separated; more readable and testable behavior. Impact: faster review, easier to locate regression.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
