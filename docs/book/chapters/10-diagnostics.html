Concrete Problem

10.1 Division test

proc safe_div(num: int, den: int) -> int {

  if den == 0 { give 0 }

  give num / den
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: den == 0. If true, the test path is executed immediately.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns 0.
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: test a nominal input then a limit input, and compare the two outputs obtained.

The intention of this step is direct: to place the error boundary as close as possible to the risky operation.

This principle avoids avoidable crashes: the division is protected before being executed.

At runtime:

  • safe_div(12,3) returns 4.
  • safe_div(12,0) returns 0 without attempting division.

Classic mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

10.2 Typed error

pick ParsePort {
  case Ok(value: int)
  case Err(code: int)
}
proc parse_port(x: int) -> ParsePort {

  if x < 0 { give Err(400) }

  if x > 65535 { give Err(422) }

  give Ok(x)
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: x < 0. If true, the test path is executed immediately.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns Err(400).
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: test a nominal input then a limit input, and compare the two outputs obtained.

The intent of this step is straightforward: encode the diagnosis in the return type, to make failures as explicit as successes.

With this model, we do not lose the error information: each case carries its associated code.

At runtime:

  • parse_port(-1) returns Err(400).
  • parse_port(8080) returns Ok(8080).
  • parse_port(70000) returns Err(422).

Classic mistakes to avoid:

  • encoding implicit conventions instead of carrying them by type.
  • mix different business cases in the same representation.
  • add variants without updating treatment points.

10.3 Technical projection

// Conversion finale vers un code de sortie
proc to_exit(p: ParsePort) -> int {

  match p {
    case Ok(_) { give 0 }
    case Err(c) { give c }
    otherwise { give 70 }
  }
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Branch selection: match p chooses a case according to the current state.
  3. Processing: Apply the operations in the written order, without skipping any implicit steps.
  4. Output produced: current path returns 0.
  5. Intermediate notion: a function is a transformation Entrée -> Sortie; its rule serves to limit ambiguities.
  6. Intermediate concept: an invariant is a property that remains true during the loop or during a pipeline of calls.

Quick check: Trace an execution with concrete values ​​(x=..., i=...) to confirm the branch actually taken.

The intention of this step is direct: to separate the business logic from its technical projection (here, the exit code).

This decoupling is important: the system policy can evolve without rewriting the analysis logic.

At runtime:

  • to_exit(Ok(_)) returns 0.
  • to_exit(Err(422)) returns 422.
  • otherwise tests a fallback code (70).

Classic mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

Extended Example

// Scenario diagnostics: execution complete et verifiable
space demo/diagnostics

form Event { code: int severity: int payload_len: int }
pick Diagnostic { case Info(code: int) case Warn(code: int) case Error(code: int) }

// Classification: mappe un événement vers un niveau explicite
proc classify(e: Event) -> Diagnostic {

  if e.code == 0 { give Diagnostic.Info(0) }

  if e.severity <= 2 { give Diagnostic.Warn(e.code) }

  give Diagnostic.Error(e.code)
}

// Redaction: borne la charge utile avant diffusion
proc redact(e: Event) -> int {

  if e.payload_len < 0 { give 81 }

  if e.payload_len > 4096 { give 82 }

  give 0
}

proc handle(e: Event) -> int {
  let r: int = redact(e)

  if r != 0 { give r }
  let d: Diagnostic = classify(e)

  match d {
    case Info(_) { give 0 }
    case Warn(_) { give 0 }
    case Error(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let e: Event = Event(17, 3, 120)

  return handle(e)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.
  • classify: reads e: Event and returns Diagnostic.
  • redact: reads e: Event and returns int.
  • handle: reads e: Event and returns int.

2. Actual execution order

  1. The program enters main.
  2. handle is called to process the next step.
  3. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: e.code == 0.
  • Test evaluated: e.severity <= 2.
  • Test evaluated: e.payload_len < 0.
  • Test evaluated: e.payload_len > 4096.
  • Test evaluated: r != 0.
  • Selection by match d: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> handle -> sortie 0.
  • Example error: entrée invalide -> handle renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Info or warn level -> output 0.
  • Business error code 17 -> exit 17.
  • Payload out of range -> exit 82.

7. Line by line (variables + values)

Convenient reading: Follow the variables in the actual order of execution, then check the observed output.

  • Entry point:
  • entry main at core/app launches the complete scenario.
  • Block functions:

  • redact reads e: Event then returns int.

  • Variables created (initial value):

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
