Concrete Problem

Real situation: Performance becomes clear when we trace each step of the calculation. The goal is to connect line of code and concrete effect on the output. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: move from passive reading to proof: same input, same branch, same expected output.

Goal

You need to be able to reread a snippet, predict its outcome, and then verify that prediction with simple execution.

Why it matters

This chapter gives you a clear understanding of Performance. You will find the framework, the invariants and the reading decisions useful in practice.

What you will do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

19.1 Establish a baseline

proc sum_loop(n: int) -> int {
  let i: int = 0
  let acc: int = 0

  // Boucle: progression controlee jusqu'a la borne
  loop {
    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= n { break }
    set acc = acc + i
    set i = i + 1
  }

  give acc
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: i >= n. If true, the test path is executed immediately.
  3. Loop: check the stopping condition and state progress in each iteration.
  4. Output produced: current path returns acc.
  5. Key concept: we separate functional correction and algorithmic cost to optimize without breaking the behavior.
  6. Key concept: comparing two variants requires the same input and a strictly equivalent output.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

The intention of this step is direct: to establish a readable baseline which will serve as a point of comparison.

This version defines both the expected result and the initial cost structure (one loop, one addition, one increment each round).

At runtime, sum_loop(4) accumulates 0+1+2+3 and returns 6.

Classic mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

19.2 Variant with filter branch

proc sum_even(n: int) -> int {
  let i: int = 0
  let acc: int = 0

  // Boucle: progression controlee jusqu'a la borne
  loop {
    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= n { break }
    if (i % 2) != 0 {
      set i = i + 1
      continue
    }
    set acc = acc + i
    set i = i + 1
  }

  give acc
}

Guided algorithmic reading:

  1. Input read: first identify the parameters and their type, these are the starting data for the calculation.
  2. Condition evaluated first: i >= n. If true, the test path is executed immediately.
  3. Loop: check the stopping condition and state progress in each iteration.
  4. Output produced: current path returns acc.
  5. Key concept: we separate functional correction and algorithmic cost to optimize without breaking the behavior.
  6. Key concept: comparing two variants requires the same input and a strictly equivalent output.

Quick check: Keep the same entry before/after modification to prove that the expected behavior is preserved.

The intention of this step is straightforward: to introduce conditional filtering to only sum even values.

The correction remains intact, but the cost profile changes: an additional branch is evaluated in each iteration.

Step by step reading for sum_even(6):

  • i=0 is even, acc=0.
  • i=1 is odd, continue.
  • i=2 is even, acc=2.
  • i=3 is odd, continue.
  • i=4 is even, acc=6.
  • i=5 is odd, continue.
  • stop on i=6, return 6.

Classic mistakes to avoid:

  • encoding implicit conventions instead of carrying them by type.
  • mix different business cases in the same representation.
  • add variants without updating treatment points.

The intention of this step is straightforward: remove the filter branch by moving directly from two to two.

Keep in mind

Each optimization preserves the result, changes one cost variable at a time and remains measurable. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

Practical quality criterion for this chapter:

  • you know how to prove that two variants calculate the same result.
  • you know how to identify what changes in the execution cost.
  • you know how to compare versions without mixing several optimizations at the same time.

Extended Example

// Scenario performance: execution complete et verifiable
space demo/performance

pick Bench { case Stable(avg: int, p95: int) case Unstable(code: int) }

proc workload(n: int) -> int {
  let i: int = 0
  let acc: int = 0

  // Boucle: progression controlee jusqu'a la borne
  loop {
    // Borne d'arret: stoppe la boucle de maniere explicite
    if i >= n { break }
    set acc = acc + (i * 5)
    set i = i + 1
  }

  give acc
}

proc sample(iter: int, size: int) -> int {
  let base: int = size * 10
  let jitter: int = iter % 9

  give base + jitter
}

// Benchmark: warmup + mesures + décision de stabilité
proc benchmark(size: int) -> Bench {

  if size <= 0 { give Bench.Unstable(41) }
  let w: int = workload(120)
  let _w: int = w
  let s1: int = sample(1, size)
  let s2: int = sample(2, size)
  let s3: int = sample(3, size)
  let s4: int = sample(4, size)
  let s5: int = sample(5, size)
  let avg: int = (s1 + s2 + s3 + s4 + s5) / 5
  let p95: int = s5

  if p95 > (avg * 2) { give Bench.Unstable(42) }

  give Bench.Stable(avg, p95)
}

// Conversion finale vers un code de sortie
proc to_exit(b: Bench) -> int {

  match b {
    case Stable(_, _) { give 0 }
    case Unstable(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let b: Bench = benchmark(600)

  return to_exit(b)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • workload: reads n: int and returns int.
  • sample: reads iter: int, size: int and returns int.
  • benchmark: reads size: int and returns Bench.
  • to_exit: reads b: Bench and returns int.

2. Actual execution order

  1. The program enters main.
  2. benchmark is called to process the next step.
  3. to_exit is called to process the next step.
  4. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: i >= n.
  • Test evaluated: size <= 0.
  • Test evaluated: p95 > (avg * 2).
  • Selection by match b: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> benchmark -> to_exit -> sortie 0.
  • Example error: entrée invalide -> benchmark renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Stable campaign -> exit 0.
  • Invalid parameter (size=0) -> output 41.
  • Excessive variance -> output 42.

Trade-offs

Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
