0. Preface

Level: Beginner

Prerequisites: no technical prerequisites; read docs/book/glossary.html for basic vocabulary. See also: docs/book/chapters/01-starting.html, docs/book/glossary.html.

Concrete Problem

Real situation: Preface is best understood by replaying the program as an executable algorithm. You read the incoming data, the condition evaluated, and then the returned value. Guiding question: Which condition is evaluated first, and what output does this decision impose?

Red Thread (Single Project)

Common thread: we keep the same mini-program to compare the effects of a modification without changing the entire context. Educational objective: Reread a block, predict the output, then confirm the prediction with a simple, repeatable execution.

Why it matters

This chapter gives you a clear understanding of Preface. You will find the framework, the invariants and the reading decisions useful in practice.

What you will do

You'll read the snippets in actual execution order, then validate the expected outputs on a nominal case and an error case.

Minimal example

Recommended first reflex: read the entries and conditions first, then only the syntactic form.

Step-by-step explanation

  1. Identify the intent of the block.
  2. Check the main condition or test.
  3. Confirm the observable output.
  4. Note how this block serves Preface throughout the chapter.

Common pitfalls

  • Read the syntax without checking the behavior.
  • Mix general rule and error cases in the same explanation.
  • Introduce an optimization before having stabilized the Preface flow.

Short exercise

Take an example from the chapter on Preface. Change a condition or input value, then check if the result remains as expected.

Summary in 5 points

  1. You know the purpose of the chapter on Preface.
  2. You know how to read an example of the chapter in a structured way.
  3. You distinguish nominal case and error case.
  4. You avoid the most common pitfalls.
  5. You can reuse these rules in the next chapter.

0.1 Understanding the promise of language

proc promise(a: int, b: int) -> int {
  give a + b
}

Reading line by line (beginner):

  1. proc promise(a: int, b: int) -> int {: the rule is defined for promise: inputs a: int, b: int and output int, it clarifies the intention before detailed reading of the body.
  2. give a + b: the branch returns a + b for the current branch, the output is explicit and easy to verify.
  3. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: without blocking test, the main branch returns a + b.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should trigger a clear validation or fallback path.

The intention of this step is direct: Vitte favors the clarity of the rule over syntactic magic. The reader must always see what is coming in, what is going out and why.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, promise(20,22)=42.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

0.2 Accept the discipline of invariants

proc bounded(x: int) -> int {

  if x < 0 { give 0 }

  if x > 100 { give 100 }

  give x
}

Reading line by line (beginner):

  1. proc bounded(x: int) -> int {: the rule is set for bounded: inputs x: int and output int, it clarifies the intention before detailed reading of the body.
  2. if x < 0 { give 0 }: this test handles the invalid case before the main path.
  3. if x > 100 { give 100 }: this test handles the invalid case before the main path.
  4. give x: the branch returns x for the current branch, the output is explicit and easy to verify.
  5. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: if x < 0 is true, the output becomes 0.
  • Nominal case: without blocking test, the main branch returns x.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should trigger a clear validation or fallback path.

The intent of this step is straightforward: An invariant is a local rule that avoids entire classes of bugs. The book emphasizes this practice in all chapters.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, bounded(-1)=0, bounded(70)=70, bounded(140)=100.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

0.3 Build a project-oriented progression

// Point d'entree du scenario
entry main at core/app {
  let v: int = bounded(140)

  return v
}

Reading line by line (beginner):

  1. entry main at core/app {: This line sets the entry point main into core/app and serves as an end-to-end executable scenario for the chapter.
  2. let v: int = bounded(140): this line creates the variable v of type int to explicitly name an intermediate step of the reasoning.
  3. return v: this line ends the execution of the current block with the code v, useful for observing the overall result of the scenario.
  4. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the main scenario ends with return v.
  • Testable observation: Running the scenario verifies the exit code v.

Quick check: invalid input should trigger a clear validation or fallback path.

The intention of this step is direct: The theory is always connected to a concrete execution flow. Each abstract notion is reduced to an observable behavior.

In a production reading, this choice reduces the mental cost: we immediately see what is guaranteed, what is refused, and where the decision is made.

At runtime, main calculates bounded(140) then finishes with 100.

Common mistakes to avoid:

  • mix orchestration and business logic in the same function.
  • duplicate a transversal rule instead of centralizing it.
  • evolve the architecture without clarifying responsibilities.

Extended Example

// Scenario preface: execution complete et verifiable
space demo/preface

form Input { id: int value: int quota: int }
pick Eval { case Accepted(score: int) case Rejected(code: int) }

proc validate(x: Input) -> Eval {

  if x.id <= 0 { give Eval.Rejected(21) }

  if x.quota < 0 { give Eval.Rejected(22) }

  if x.value < 0 { give Eval.Rejected(23) }

  give Eval.Accepted(x.value)
}

proc transform(score: int, quota: int) -> int {
  let capped: int = score
  if capped > quota { set capped = quota }

  if capped < 0 { give 0 }

  give capped * 2
}

proc decide(r: Eval, quota: int) -> Eval {

  match r {
    case Accepted(s) {
      let out: int = transform(s, quota)

      if out >= 10 { give Eval.Accepted(out) }

      give Eval.Rejected(31)
    }
    case Rejected(c) { give Eval.Rejected(c) }
    otherwise { give Eval.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(r: Eval) -> int {

  match r {
    case Accepted(_) { give 0 }
    case Rejected(code) { give code }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let x: Input = Input(1, 8, 9)
  let v: Eval = validate(x)
  let d: Eval = decide(v, x.quota)

  return to_exit(d)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.
  • validate: reads x: Input and returns Eval.
  • transform: reads score: int, quota: int and returns int.
  • decide: reads r: Eval, quota: int and returns Eval.
  • to_exit: reads r: Eval and returns int.

2. Actual execution order

  1. The program enters main.
  2. validate is called to process the next step.
  3. decide is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: x.id <= 0.
  • Test evaluated: x.quota < 0.
  • Test evaluated: x.value < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 0.
  • Test evaluated: out >= 10.
  • Selection by match r: the path depends on the state received.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> validate -> decide -> to_exit -> sortie 0.
  • Example error: entrée invalide -> validate renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Nominal case -> output 0.
  • Strict quota case -> stable behavior.
  • Invalid case id<=0 -> output 21.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Mini Case Study (Before / After)

Before: mixed business logic and technical output, expensive diagnosis. After: entrance tests, business decision, final screening separated; more readable and testable behavior. Impact: faster review, easier to locate regression.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
