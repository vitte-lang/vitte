27.1 Construct a valid procedure statement

proc add(a: int, b: int) -> int {
  give a + b
}

Reading line by line (beginner):

  1. proc add(a: int, b: int) -> int {: the rule is defined for add: inputs a: int, b: int and output int, it clarifies the intention before detailed reading of the body.
  2. give a + b: the branch returns a + b for the current branch, the output is explicit and easy to verify.
  3. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: without blocking test, the main branch returns a + b.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should be rejected before the main path.

The intent of this step is straightforward: to validate the canonical form of a procedure statement.

Grammar reading:

  • proc header with name and typed parameters.
  • explicit return type -> int.
  • delimited block containing a return instruction.

At runtime, add(1,2)=3.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

27.2 Constructing an explicit program entry

// Point d'entree du scenario
entry main at core/app {
  return 0
}

Reading line by line (beginner):

  1. entry main at core/app {: This line sets the entry point main into core/app and serves as an end-to-end executable scenario for the chapter.
  2. return 0: this line ends the execution of the current block with the code 0, useful for observing the overall result of the scenario.
  3. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the main scenario ends with return 0.
  • Testable observation: Running the scenario verifies the exit code 0.

Quick check: invalid input should be rejected before the main path.

The intent of this step is straightforward: to make the program's entry point explicit.

The form entry.. at.. fixes both the logical name and the attachment module.

When executed, the program immediately exits with the code 0.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

27.3 Constructing a match with fallback (patterns)

pick Resp { case Ok, case Err }
proc to_code(r: Resp) -> int {

  match r {
    case Ok { give 0 }
    otherwise { give 1 }
  }
}

Reading line by line (beginner):

  1. pick Resp { case Ok, case Err } this line defines a closed type to have valid patterns in match.
  2. proc to_code(r: Resp) -> int { this line sets a simple rule: variant entry, code exit.
  3. match r { this line starts dispatch by pattern.
  4. case Ok { give 0 } this line explicitly covers the Ok pattern.
  5. otherwise { give 1 } this line covers the fallback.
  6. }: this brace closes the block.
  7. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the flow follows the main branch and produces a stable output.
  • Testable observation: forcing the case Ok confirms the expected branch.

Quick check: invalid input should be rejected before the main path.

The intention of this step is direct: to build a readable and exhaustive match, conforming to the grammar patterns.

Here, otherwise explicitly closes the decision tree, which avoids flow control holes.

At runtime, to_code(Ok)=0 and to_code(Err)=1.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

27.4 Read the form foo[T](..) without overinterpreting

proc id[T](x: T) -> T {
  give x
}

proc main() -> int {
  let a = id[int](1)
  let i = 0
  let b = arr[i](1)

  give 0
}

Useful grammar reading:

  1. id[int](1) can be read as an explicit generic call.
  2. arr[i](1) should not be reread too quickly as a generic call.
  3. The same surface foo[..](..) therefore carries a real ambiguity.

Current parser rule:

  • if the bracketed content is read as a list of unambiguous types, the form becomes an explicit wildcard call
  • otherwise, reading remains index_suffix then call_suffix

Current consequences:

  • id[int](1) is accepted as an explicit generic call
  • Public[int](1) and facade_mod.Public[int](1) follow the same logic
  • id[i](1) remains an indexing followed by call

This terminal is voluntary. It documents actual behavior without pretending that any foo[..](..) form is already stabilized.

Keep in mind

Critical syntactic forms are controlled, which reduces structural errors before type-checking. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

Practical quality criterion for this chapter:

  • you know how to immediately recognize an invalid syntactic form.
  • you know how to distinguish grammatical errors and type errors.

Extended Example

// Scenario grammaire: execution complete et verifiable
space demo/grammaire

form SourceUnit { bytes: int lines: int tokens_hint: int }
pick ParseState { case Parsed(nodes: int) case Failed(code: int) }

// Scan: transforme l'entrée brute en signal exploitable
proc scan(u: SourceUnit) -> int {

  if u.bytes <= 0 { give 0 }

  if u.lines <= 0 { give 0 }

  give (u.tokens_hint + u.lines)
}

// Parse: construit un état syntaxique stable
proc parse(token_count: int) -> ParseState {

  if token_count == 0 { give ParseState.Failed(101) }

  if token_count < 4 { give ParseState.Failed(102) }

  give ParseState.Parsed(token_count)
}

proc validate_structure(nodes: int) -> int {

  if nodes <= 0 { give 201 }

  if nodes > 200000 { give 202 }

  give 0
}

// Conversion finale vers un code de sortie
proc to_exit(p: ParseState) -> int {

  match p {
    case Parsed(n) {
      let v: int = validate_structure(n)

      if v != 0 { give v }

      give 0
    }
    case Failed(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let u: SourceUnit = SourceUnit(120, 12, 18)
  let t: int = scan(u)
  let p: ParseState = parse(t)

  return to_exit(p)
}

Detailed explanation of the big block

You read this big block as a complete scenario: data preparation, processing, then final output.

1. Role of each party

  • Starting point: entry main at core/app.
  • scan: reads u: SourceUnit and returns int.
  • parse: reads token_count: int and returns ParseState.
  • validate_structure: reads nodes: int and returns int.
  • to_exit: reads p: ParseState and returns int.

2. Actual execution order

  1. The program enters main.
  2. scan is called to process the next step.
  3. parse is called to process the next step.
  4. to_exit is called to process the next step.
  5. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: u.bytes <= 0.
  • Test evaluated: u.lines <= 0.
  • Test evaluated: token_count == 0.
  • Test evaluated: token_count < 4.
  • Test evaluated: nodes <= 0.
  • Test evaluated: nodes > 200000.
  • Selection by match p: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> scan -> parse -> to_exit -> sortie 0.
  • Example error: entrée invalide -> scan renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Valid unit -> output 0.
  • Empty input (bytes=0) -> output 101.
  • Oversized structure -> exit 202.

Trade-offs

Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

At this point you should know:

  • explain the input -> decision -> output flow without ambiguity,
  • isolate a real error case and predict its exit,
  • identify where to add a test without breaking the nominal.

Why This Error Happens In Prod

Common cause: partially valid input, implicit assumption in a branch, then projecting output too late. Symptom: correct behavior in nominal but unstable under load or incomplete data. Useful action: trace the actual input, replay the error case, lock the test at the correct level.

Mini Case Study (Before / After)

Before: mixed business logic and technical output, expensive diagnosis. After: entrance tests, business decision, final screening separated; more readable and testable behavior. Impact: faster review, easier to locate regression.

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
