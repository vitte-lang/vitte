28.1 Naming procedures according to their intent

proc parse_port(x: int) -> int {

  if x < 0 { give -1 }

  if x > 65535 { give -1 }

  give x
}

Reading line by line (beginner):

  1. proc parse_port(x: int) -> int {: the rule is defined for parse_port: inputs x: int and output int, it clarifies the intention before detailed reading of the body.
  2. if x < 0 { give -1 }: this test handles the invalid case before the main path.
  3. if x > 65535 { give -1 }: this test handles the invalid case before the main path.
  4. give x: the branch returns x for the current branch, the output is explicit and easy to verify.
  5. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: if x < 0 is true, the output becomes -1.
  • Nominal case: without blocking test, the main branch returns x.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should be rejected before the main path.

The intention of this step is direct: to convey the business intention by the name of the procedure.

The verb parse signals a validation rule, and the leading tests make the flow immediate to replay.

At runtime, parse_port(8080)=8080, parse_port(70000)=-1.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

28.2 Naming structures according to domain

form User {
  id: int
  name: string
}

Reading line by line (beginner):

  1. form User {: this line opens the User structure which groups coherent data under the same business name, useful for keeping a stable vocabulary.
  2. id: int: this line declares the field id with the type int, which documents its role and limits handling errors.
  3. name: string: this line declares the field name with the type string, which documents its role and limits handling errors.
  4. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: the flow follows the main branch and produces a stable output.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should be rejected before the main path.

The intent of this step is straightforward: to use structure and field names that describe the domain, not the implementation.

The diagram becomes self-documenting: the structure reads like a mini business specification.

At runtime, this statement does not execute anything, but it sets a clear data rule at compile-time.

Common mistakes to avoid:

  • encoding implicit conventions instead of carrying them by type.
  • mix different business cases in the same representation.
  • add variants without updating treatment points.

28.3 Naming predicates in explicit Boolean

proc is_admin(role: int) -> bool {
  give role == 9
}

Reading line by line (beginner):

  1. proc is_admin(role: int) -> bool {: the rule is set for is_admin: inputs role: int and output bool, it clarifies the intention before detailed reading of the body.
  2. give role == 9: the branch returns role == 9 for the current branch, the output is explicit and easy to verify.
  3. }: this brace closes the block.

Input -> output (to be checked):

  • Error case: an explicit test of the block handles out-of-rule entries before the nominal path.
  • Nominal case: without blocking test, the main branch returns role == 9.
  • Testable observation: repeating the same input should reproduce exactly the same output.

Quick check: invalid input should be rejected before the main path.

The intention of this step is straightforward: to make the predicates immediately recognizable in the conditions.

The is_ prefix reduces the cognitive load: in an if, we directly read a Boolean question.

At runtime, is_admin(9)=true and is_admin(1)=false.

Common mistakes to avoid:

  • accumulating special cases without clarifying intent.
  • introduce complexity before stabilizing the behavior.
  • leave implicit decisions that slow down rereading.

Keep in mind

Names express intent, tests lead, and Boolean predicates are instantly recognizable. This chapter should leave you with a stable reading grid: visible intention, explicit rule, and observable behavior from start to finish.

  • you know how to distinguish a Boolean predicate from a transformation.

Extended Example

// Scenario conventions: execution complete et verifiable
space demo/conventions

form ModuleSpec { name_len: int major: int minor: int }
pick Resolve { case Loaded(path_code: int) case Missing(code: int) }

proc normalize_len(n: int) -> int {

  if n <= 0 { give 0 }

  give n
}

// Résolution: valide la référence et produit une décision de chargement
proc resolve(m: ModuleSpec) -> Resolve {
  let n: int = normalize_len(m.name_len)

  if n == 0 { give Resolve.Missing(71) }

  if m.major <= 0 { give Resolve.Missing(72) }

  if m.minor < 0 { give Resolve.Missing(73) }
  let code: int = 100 + (m.major * 10) + m.minor

  give Resolve.Loaded(code)
}

// Conversion finale vers un code de sortie
proc to_exit(r: Resolve) -> int {

  match r {
    case Loaded(_) { give 0 }
    case Missing(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let m: ModuleSpec = ModuleSpec(8, 1, 2)
  let r: Resolve = resolve(m)

  return to_exit(r)
}

Detailed explanation of the big block

This large block shows an entire program, not an isolated extract: we follow the flow from start to finish.

1. Role of each party

  • Starting point: entry main at core/app.
  • normalize_len: reads n: int and returns int.
  • resolve: reads m: ModuleSpec and returns Resolve.
  • to_exit: reads r: Resolve and returns int.

2. Actual execution order

  1. The program enters main.
  2. resolve is called to process the next step.
  3. to_exit is called to process the next step.
  4. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: n <= 0.
  • Test evaluated: n == 0.
  • Test evaluated: m.major <= 0.
  • Test evaluated: m.minor < 0.
  • Selection by match r: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> resolve -> to_exit -> sortie 0.
  • Example error: entrée invalide -> resolve renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Design Notes

  • The snippet favors explicit boundaries rather than minimalist code.
  • Tests are placed early to reduce the cost of diagnosis.
  • The output is projected at the end of the flow to keep the business independent of transport.

Actual error case:

  • Degraded or incomplete input: the test must cut the flow early with an explicit exit.

To test:

  • Valid specification -> output 0.
  • Invalid name -> exit 71.
  • Invalid version -> output 72 or 73.

Trade-offs

ConstraintOption AOption BRecommended decision
Priority readabilityExplicit branchesCompact codeIf the team maintains the code for a long time
Critical performanceTargeted specializationGeneralizationA if profiling confirms the gain
Rapid evolutionStrict contractsImplicit conventionsA to reduce regressions

Decision According to Constraint

  • If the dominant constraint is safety: validate early, fail explicitly.
  • If the dominant constraint is latency: measure first, optimize later.
  • If the dominant constraint is scalability: isolate orchestration, decisions and output conversion.

Checkpoint

What I would do in code review

  1. Verify that input tests are placed before sensitive operations.
  2. Verify that the business decision is separated from the output projection.
  3. Verify a nominal test and a limit test that are actually executable.
