3. Structure of a project

Concrete Problem

A project quickly becomes confusing when responsibilities are mixed. The goal of this chapter: organize the code to immediately know where to read, where to modify, and where to test.

Red Thread (Single Project)

We keep the same scenario to see the impact of a good structure. You will follow a simple path: input -> processing -> result.

Goal

We divide the project into 3 zones:

  1. domain: business data
  2. service: calculation rules
  3. io: launch and release

Recommended feed: io -> service -> domain.

3.1 Domain: define business data

space app/domain

form Ticket {
  id: int
  priority: int
}

Direct reading:

  1. Ticket describes data useful to the business.
  2. no execution logic here.
  3. this module remains stable even if the application changes channel (CLI, HTTP, job).

3.2 Service: centralize business rules

space app/service

pull app/domain as d

proc is_critical(t: d.Ticket) -> bool {
  give t.priority >= 9
}

Direct reading:

  1. service uses the type of domain.
  2. the business rule is written only once.
  3. the output is boolean: true or false.

Example:

  • priority 9 -> true
  • priority 4 -> false

3.3 IO: start the stream and produce output

space app/io

pull app/domain as d
pull app/service as s

// Point d'entree du scenario
entry main at core/app {
  let t: d.Ticket = d.Ticket(1, 9)
  let critical: bool = s.is_critical(t)
  let code: int = critical as int

  return code
}

Direct reading:

  1. main constructs an entry.
  2. main calls the service.
  3. main converts the result to an exit code.

To avoid in io:

  • write business rules here
  • duplicate the logic already present in service

3.4 Signals of poor structure

  1. a business rule is written in entry
  2. service depends on io
  3. a business type contains technical output details

If any of these signals appear, maintenance will cost more.

Quick structure test (30 seconds)

For each file, ask 3 questions:

  1. Is this business data? -> domain
  2. Is this a business rule? -> service
  3. is this launch/release? -> io

If a file answers several questions, it must be split.

3.6 Checklist before commit

  1. is the direction of dependencies io -> service -> domain ?
  2. are the rules in service ?
  3. are business types in domain?
  4. entry is it short and readable?

Extended Example

// Scenario projet: execution complete et verifiable
space demo/projet

form Request { id: int amount: int quota: int }
pick Result { case Accepted(total: int) case Rejected(code: int) }

// Entrée applicative: validation des invariants de requête
proc parse_request(r: Request) -> Result {

  if r.id <= 0 { give Result.Rejected(91) }

  if r.quota < 0 { give Result.Rejected(92) }

  if r.amount < 0 { give Result.Rejected(93) }

  give Result.Accepted(r.amount)
}

// Politique métier: applique les règles de décision
proc apply_policy(total: int, quota: int) -> Result {
  let capped: int = total
  if capped > quota { set capped = quota }

  if capped < 5 { give Result.Rejected(94) }

  give Result.Accepted(capped)
}

// Persistance simulée: matérialise un résultat sans I/O réel
proc persist_sim(x: Result) -> Result {

  match x {
    case Accepted(v) {
      if v % 13 == 0 { give Result.Rejected(95) }

      give Result.Accepted(v)
    }
    case Rejected(c) { give Result.Rejected(c) }
    otherwise { give Result.Rejected(70) }
  }
}

// Conversion finale vers un code de sortie
proc to_exit(x: Result) -> int {

  match x {
    case Accepted(_) { give 0 }
    case Rejected(c) { give c }
    otherwise { give 70 }
  }
}

// Point d'entree du scenario
entry main at core/app {
  let req: Request = Request(7, 12, 15)
  let p: Result = parse_request(req)
  let d: Result = apply_policy(12, req.quota)
  let s: Result = persist_sim(d)
  let _probe: int = to_exit(p)

  return to_exit(s)
}

Detailed explanation of the big block

Here the goal is to understand the actual path of the program, line by line, to the exit code.

1. Role of each party

  • Starting point: entry main at core/app.
  • parse_request: reads r: Request and returns Result.
  • apply_policy: reads total: int, quota: int and returns Result.
  • persist_sim: reads x: Result and returns Result.
  • to_exit: reads x: Result and returns int.

2. Actual execution order

  1. The program enters main.
  2. parse_request is called to process the next step.
  3. apply_policy is called to process the next step.
  4. persist_sim is called to process the next step.
  5. to_exit is called to process the next step.
  6. The final value is converted to process output (return ...).

3. Tests that change the path

  • Test evaluated: r.id <= 0.
  • Test evaluated: r.quota < 0.
  • Test evaluated: r.amount < 0.
  • Test evaluated: capped > quota.
  • Test evaluated: capped < 5.
  • Test evaluated: v % 13 == 0.
  • Selection by match x: the path depends on the state received.
  • Selection by match x: the path depends on the state received.

4. Quick trace with values

  • Nominal example: entrée valide -> parse_request -> apply_policy -> persist_sim -> to_exit -> sortie 0.
  • Example error: entrée invalide -> parse_request renvoie un code d'erreur -> sortie non nulle.

5. Why this division is useful

  • You test each function on its own, then the entire flow.
  • You know where to change a rule without breaking the entire program.
  • You can explain the output by simply following the calls.

6. Quick Check

  1. Retry with a normal entry and note the exit.
  2. Restart with invalid input and check the error code.
  3. Confirm that the same input always gives the same output.

Play full stream

Order of execution:

  1. main builds req.
  2. parse_request controls the data.
  3. apply_policy applies the business rule.
  4. persist_sim simulates final processing.
  5. to_exit produces the exit code.

Normal execution:

  • entry: Request(7, 12, 15)
  • final output: 0

Running Error 1:

  • entry with id = 0
  • final output: 91

Running Error 2:

  • entry with amount = 3
  • final output: 94

7. Line by line (variables + values)

Convenient reading: Follow the variables in the actual order of execution, then check the observed output.

  • Entry point:
  • entry main at core/app launches the complete scenario.
  • Block functions:
  • parse_request reads r: Request then returns Result.
  • apply_policy reads total: int, quota: int then returns Result.
  • persist_sim reads x: Result then returns Result.
  • to_exit reads x: Result then returns int.
  • Variables created (initial value):
  • capped: int starts with total.
  • req: Request starts with Request(7, 12, 15).
  • p: Result starts with parse_request(req).
  • d: Result starts with apply_policy(12, req.quota).
  • s: Result starts with persist_sim(d).
  • _probe: int starts with to_exit(p).
  • Variables changed during processing:
  • capped is updated with quota.
  • Conditions that change the path:
  • if r.id <= 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if r.quota < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if r.amount < 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if capped > quota is true: early exit or dedicated branch; otherwise: the flow continues.
  • if capped < 5 is true: early exit or dedicated branch; otherwise: the flow continues.
  • if v % 13 == 0 is true: early exit or dedicated branch; otherwise: the flow continues.
  • Nominal trace (example values):
  • initialization: capped=total -> req=Request(7, 12, 15) -> p=parse_request(req) -> d=apply_policy(12, req.quota)
  • sequence: parse_request -> apply_policy -> persist_sim -> to_exit
  • final output on this path: to_exit(s).
  • Error trace (example values):
  • if r.id <= 0 becomes true, the function returns Result.Rejected(91).
  • Quick check:
  • restart with normal input and note the output,
  • restart with an invalid entry and note the error code,
  • confirm that the same input always produces the same output.

Checkpoint

At this point you should know:

  • place a file in the right place,
  • follow the flow of a project without confusion,
  • explain the output from the called functions.

Mini Case Study (Before / After)

Before: a single file mixes input, rules and output. After: domain describes the data, service decides, io launches. Result: faster reading, simpler correction, more stable tests.

What I would do in code review

  1. check the direction of dependencies (io -> service -> domain)
  2. check that the rules are not in entry
  3. check a normal scenario + an error scenario
