Source of truth

Start with the first code, then read the corresponding stable file in docs/book/errors.html. To confirm the meaning of the message, go through docs/book/cli.html and the order vitte explain <code>.

Official sequence

  • vitte explain <code>
  • vitte check --diag-code-only
  • vitte mod doctor --fix
  • vitte check --diag-json-pretty

Frequent codes

  • E0001 - expected identifier
  • E0013 - unknown identifier
  • E0019 - proc requires a body
  • E0024 - select requires at least one when branch

What to do first

  • Read the first code, not the whole waterfall.
  • Open the corresponding file in docs/book/errors.html.
  • Throw vitte explain <code> to confirm the official meaning.
  • Correct locally, then restart on a minimal case.

Quick index

  • parse : E0001 to E0008.
  • resolve : E0009 to E0025.
  • IR : E0015 to E0028.

Practical reading

Diagnostics should be read as a contract, not as a wall of text. The code says what failed, the book says what the code means, and the CLI explains how to inspect it. If those three disagree, the first place to trust is the current source, then the dedicated chapter, then the command output.

For a fast triage, use this order:

  1. Identify the code.
  2. Open the matching entry in docs/book/errors.html.
  3. Run vitte explain <code> to confirm the current text.
  4. Reduce the repro to the smallest snippet that still fails.
  5. Check whether the issue belongs to parsing, resolution, lowering, or a post-parse rule.

That sequence helps contributors stay precise. It also keeps bug reports useful, because the report names the code, the layer, the smallest repro, and the expected behavior instead of only describing the symptom.

If you need a broader starting point, read Documentation for the compiler surface and Source for the implementation path.
