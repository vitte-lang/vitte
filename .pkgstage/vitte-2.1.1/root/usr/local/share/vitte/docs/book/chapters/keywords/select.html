Keyword select

Level: Intermediate.

Quick reading

Concrete reference: select is used to decide an execution path in a readable and verifiable manner. Use select when it is used to drive a control decision; avoid it if it makes no observable difference on the branch or output.

Why (job)

In practice, select should make the rule clearer, not noisier. If removing it changes neither the meaning nor the outcome, prefer the simpler form.

Definition

select applies a sequence of when clauses to an expression, with otherwise optional.

Syntax

Canonical form: select expr {when_stmt} [otherwise block].

Algorithmic reading

Read it in context: find where select appears, then confirm the constraint it adds.

Nominal example

Input:

  • Selection on pattern variants.
// Exemple concret: cas nominal puis cas invalide

pick Status { case Ok, case NotFound, case Fail }

proc to_code(status: Status) -> int {
  select status
  when Ok { give 200 }
  when NotFound { give 404 }
  otherwise { give 500 }
}

Observed result:

  • The chosen branch depends on the status value.

Invalid example

Input:

  • Selection expression missing.
// Exemple concret: cas nominal puis cas invalide
select
when Ok { give 200 }
# invalide: `select` exige une expression.

Observed result:

  • The parser rejects the instruction.

Expected diagnosis:

  • Code: VITTE-XXXX (or corresponding E000X compiler code).
  • Position: ligne 1, colonne 1 (adjust according to actual snippet).
  • Message: stable, actionable correction.

Quick refactor

Before:

// Exemple concret: cas nominal puis cas invalide
# usage fragile à corriger

After:

// Exemple concret: cas nominal puis cas invalide
# usage clair et testable

Pitfalls

  • Use literals where a pattern is expected.
  • Forget otherwise for a closed flow.
  • Mixing select and match in the same level for no reason.

When to use it / When to avoid it

  • When to use it: to express readable multi-branch choices.
  • When to avoid it: for simple binary cases (if/else).
