Keyword all

Level: Intermediate.

Quick reading

all exports the full public surface of the current module. Use it when the module is meant to be a facade; avoid it when a shorter export list keeps the contract tighter.

Why (job)

In practice, all is a convenience for module authors who want the entire declared surface to stay visible. The tradeoff is scope: a broader export is easier to consume, but harder to audit when the module grows.

Definition

all is the share-selector that tells share to export every eligible symbol from the current module.

Syntax

Canonical form: share all.

Nominal example

Input:

  • A module that intentionally exposes its whole surface.
space demo/api

proc open() -> int {
  give 1
}

proc close() -> int {
  give 0
}

share all

Observed result:

  • Both procedures stay visible to downstream imports.

Invalid example

Input:

  • A module that forgets the selector after share.
space demo/api

proc open() -> int {
  give 1
}

share

Observed result:

  • The parser rejects the declaration because the export target is incomplete.

Expected diagnosis:

  • Code: VITTE-XXXX (or the matching compiler error code).
  • Position: ligne 7, colonne 1.
  • Message: a short correction that points to share all or to an explicit identifier list.

Quick refactor

Before:

share foo, bar, baz, qux

After:

share all

Pitfalls

  • Using all when only a small export list is needed.
  • Assuming all is a compile-time alias for every module shape.
  • Letting the exported surface grow without a review point.

When to use it / When to avoid it

  • When to use it: when the module is intentionally a facade or re-export hub.
  • When to avoid it: when the public API should stay explicit and narrow.
